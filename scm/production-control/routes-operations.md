# <a name="routes-and-operations"></a>Маршрутов и операций

В этом разделе представлена информация о маршрутах и операциях. Маршрут определяет процесс для производства продукта или варианта продукта. Описывает в производственном процессе и порядок, в котором следует выполнить следующие действия в каждый этап (операцию). Для каждого шага маршрута определяет ресурсы необходимые операции, время настройки и время выполнения учета и расчета затрат.

<a name="overview"></a>Обзор
--------

Маршрут описывает последовательность операций, необходимое для производства продукта или варианта продукта. Для каждой операции маршрута также определяет операционным ресурсам, которые являются обязательными, время, необходимое для настройки и выполнения операции и расчета затрат. Можно использовать тот же маршрут для создания нескольких продуктов или можно определить уникальный маршрут для каждого продукта или варианта продукта. Может иметь несколько маршрутов для того же продукта. В этом случае маршрут, используемый зависит от таких факторов, как количество, которое должно быть произведено. Определение маршрута в Microsoft Dynamics 365 для операций состоит из четырех отдельных элементов, которые вместе описывают производственного процесса:

-   **Маршрут** — маршрут определяет структуру производственного процесса. Другими словами он определяет порядок операций.
-   **Операции** — операции определяет указанный этап в маршруте, таких как **сборки**. Эта же операция может произойти в нескольких маршрутов и могут иметь различные операции.
-   **Связь операции** — связь операции определяет рабочих свойств операции, например, время настройки и время выполнения, категории затрат, параметры потребления и потребности в ресурсах. Связь операции позволяет рабочих свойств выполнения операции, зависят от используемой операции маршрута или продуктов произведенных.
-   **Версия маршрута** — версии маршрута определяет маршрут, используемый для производства продукта или варианта продукта. Версии маршрута позволяют маршруты должны быть повторно использован для продуктов или меняется со временем. Они также позволяют разными маршрутами, которые должны использоваться для производства одного продукта. В этом случае маршрут, используемый зависит от таких факторов, как расположение или количество, которое должно быть произведено.

## <a name="routes"></a>Маршруты
Маршрут описывает последовательность операций, используемый для производства продукта или варианта продукта. Каждой операции присвоен номер операции и последующая операция. Порядок операций формирует сеть маршрута, могут быть представлены направить диаграмму, которая имеет одну или несколько точек начальной и конечной точки. В Dynamics 365 для операций маршруты различаются в зависимости от типа структуры. Существует два типа маршрутов: простой маршрутов и маршрутов сетей. В параметры управления производством можно указать, может ли использоваться только простые маршрутов или использование более сложных маршрутными сетями.

### <a name="simple-routes"></a>Простое маршрутов

Простой путь последовательный и имеется только одна начальную точку для маршрута.  

[![Простой путь](./media/routes-and-operations-1-simple-route.png)](./media/routes-and-operations-1-simple-route.png)  

При включении только простые маршруты в производственных параметрах управления Dynamics 365 для операций автоматически создает номера операций (10, 20, 30 и т. д.) при определении маршрута.

### <a name="route-networks"></a>Маршрутными сетями

При включении более сложные маршрутными сетями в параметры управления производством можно определить маршруты, которые имеют несколько начальных точек и операций, которые могут выполняться одновременно.  

[![Route network](./media/routes-and-operations-2-route-network.png)](./media/routes-and-operations-2-route-network.png)  

**Примечания.**

-   Каждая операция может иметь только один последователь операции и всего маршрута должна быть завершена в одной операции.
-   Нет никакой гарантии, что несколько операций, которые имеют ту же операцию последующий элемент (например, операции 30 и 40 на приведенном выше рисунке) действительно будет выполнено в параллельном режиме. Сведения о доступности и мощности ресурсов можно заблокировать ограничения для того, что операции планируются.
-   0 (ноль) нельзя использовать в качестве номера операции. Этот номер будет зарезервировано и используется для указания, что последней операции в маршруте имеется не последующая операция.

### <a name="parallel-operations"></a>Параллельные операции

В некоторых случаях сочетание нескольких операционных ресурсов, имеющие различные характеристики нужно выполнить операцию. Например сборки операции может потребоваться машина, инструмент и одного работника на каждые два компьютера для наблюдения за операции. В этом примере могут быть смоделированы с использованием параллелизма, где используется в качестве первичной операции одной операции, а остальные — дополнительный.  

[![Маршрут с первичной и вторичной операций](./media/routes-and-operations-3-parallel-operations.png)](./media/routes-and-operations-3-parallel-operations.png)  

Как правило основная операция представляет минимальный ресурс и определяет время выполнения для вторичных операций. Однако во время планирования, включающий ограничения по мощности, ресурсы, которые запланированы для первичной операции и вторичные операции должны быть доступны и иметь свободные ресурсы в то же время.  

Первичной операцией и вторичные операции должны иметь один и тот же номер операции (30 на приведенном выше рисунке).  

В предыдущем примере потребность в ресурсах для первичной операции (30) — это компьютер, тогда как требования к ресурсам для вторичных операций (30' и 30''), средство и работника. 50% нагрузки гарантирует, что запланированное работника можно наблюдать за двумя компьютерами одновременно.

### <a name="approval-of-routes"></a>Утверждение маршрутов

Маршрут следует утвердить, прежде чем его можно использовать в процессе планирования и последовательно выбрав пункты производство. Утверждение указывает на завершение разработки маршрута. Таким же выпуска продукта или варианта выпущенный продукт может иметь несколько утвержденных маршрутов. Как правило утверждение маршрута происходит при первой соответствующего маршрута является утвержденной. Однако некоторые бизнес-сценариев, маршрут и версию маршрута утверждения, отдельных действий, которые может включать в себя другой процесс владельцев.  

Каждый маршрут может быть утвержденных и неутвержденных отдельно. Тем не менее Обратите внимание, что когда неутвержденных маршрут, все связанные версии маршрута можно отменить. Параметры управления производством можно указать ли маршруты могут быть неутвержденных и ли утвержденных маршрутов можно изменить.  

Если необходимо вести журнал, записей, которые утверждает каждый маршрут, можно требуются электронные подписи для утверждения маршрута. Пользователи затем необходимо подтвердить свою подлинность с помощью [электронной подписи](/dynamics365/operations/organization-administration/electronic-signature-overview).

## <a name="operations"></a>Операции
Операция, которая шаг в жизненном цикле производства. В Dynamics 365 для операций каждая операция имеет идентификатор и простое описание. Следующие таблицы показывают типичные примеры операций из мастерскую.

| Операция  | описание        |
|------------|--------------------|
| PipeCut    | Снижение канала       |

Операции, например, время настройки и время выполнения, потребности в ресурсах, сведения о затратах и расчет потребления рабочих свойств задаются на связь операции. (Дополнительные сведения о связях см. в следующем разделе.)

## <a name="operation-relations"></a>Связи операции
Связь операции сохраняются рабочих свойств следующие операции:

-   Категории затрат
-   Параметры потребления
-   Время обработки
-   Количество для обработки
-   Потребности ресурса
-   Примечания и инструкции

Можно определить несколько связей операций для той же операции. Тем не менее каждая связь операции относятся к одной операции и сохраняет свойства, характерные для маршрута, выпущенный продукт или набор используемых продуктов, которые относятся к группе номенклатур. Таким образом эта же операция используется в несколько маршрутов с различными свойствами работоспособными. Кроме того при использовании стандартных операций, которые имеют одинаковые свойства работоспособными, независимо от того, маршрут, используемый и продукт, производимый можно более легко добиться основные данные. Связи операции дают большую гибкость при определении свои маршруты. Кроме того позволяет определять свойства по умолчанию помогает сократить объем основные данные, которые необходимо поддерживать. Однако эта гибкость также означает, что вы должны быть осведомлены об изменяемых связь операции в контексте.  

**Примечание:** поскольку рабочих свойств сохраняются в связи операции по операциям маршрута, все вхождения одной и той же операции (например, сборка) имеют одинаковые время настройки, время выполнения, потребности в ресурсах и т. д. Таким образом Если два вхождения операции необходимо на одном маршруте, но имеют разное время выполнения, необходимо создать две отдельные операции, такие как сборка_1 и Assembly2.

### <a name="modifying-product-specific-routes"></a>Изменение маршрутов для конкретного продукта

При открытии **маршрут** со страницы **запущено подробные сведения о** странице отображаются те версии маршрута, связанные с выбранной выпущенный продукт. В этом контексте для каждой операции Dynamics 365 for Operations указывает рабочих свойств из связи операции, наилучшим образом соответствующее версии маршрута. Можно заметить, что список операций включает в себя **код номенклатуры** и **код маршрута** свойства из связь операции. Таким образом можно определить, какие связи операции отображается.  

На **маршрут** страницу, можно изменить свойства оперативную операции, например времени выполнения или категории затрат. Изменения сохраняются на связь операции для маршрута и выпущенный продукт, приведенные в текущей версии маршрута. Если связь операции, которая отображается не конкретной маршрута и выпущенный продукт, прежде чем изменения сохраняются, система создает копию связи операции. Эта копия *-* конкретной маршрута и выпущенный продукт. Таким образом изменения не влияют на другие маршруты или выпущенные продукты. Чтобы проверить, какие связи операции изменяется на **маршрут** страницы, просмотрите **код номенклатуры** и **код маршрута** поля.  

Можно также вручную создать операцию, предназначенный для маршрута и выпущенного продукта с помощью **копировать и редактировать связь** функции.  

**Примечание:** при добавлении новой операции с маршрутом в **маршрут**, связь операции создается страница только для текущего выпущенного продукта. Таким образом Если маршрут используется также для создания других запущенных в производство продуктов, связь применяемой операции, не существуют в течение этих запущенных в производство продуктов и маршрут больше не может использоваться для этих продуктов, запущенных в производство.

### <a name="maintaining-operation-relations-per-route"></a>Ведение связей операций на маршруте

При открытии **данных маршрута** со страницы **маршруты** страницы списка отображается список связей операций, которые применяются для выбранного маршрута. Таким образом вы можете легко проверить оперативную свойства, используемые для каких продуктов. Можно изменить значения по умолчанию и значения свойств конкретного продукта.  

Если добавить новую связь операции на **данных маршрута** страницы, **код маршрута** поля автоматически устанавливается значение **маршрут**и **связь маршрута** установлено значение номер текущего маршрута.

### <a name="maintaining-operation-relations-per-operation"></a>Ведение связей операций для операции

От **операции** страницы, можно открыть **связей операции** страницы. На этой странице можно изменять все связи операции для определенной операции. Можно даже изменить связи операции, которые содержат значения по умолчанию.  

Если организация использует стандартные операции, а рабочие параметры, одинаковы для всех продуктов и процессов, **связей операции** страница предоставляет удобный способ поддержки рабочих свойств по умолчанию из этих операций.

### <a name="applying-operation-relations"></a>Применение связей операции

В некоторых случаях Dynamics 365 для операций необходимо найти рабочих свойств для операции. Например при создании заказа на покупку, рабочих свойств каждой операции должны быть скопированы из связей операции для маршрута производства. В таких случаях Dynamics 365 for Operations выполняет соответствующие операции отношения из наиболее конкретного комбинации для минимальная конкретная комбинация.  

Когда Dynamics 365 для операций поиска наиболее подходящих связь операции для используемого продукта связи операций, соответствующий идентификатор выпущенный продукт предпочтительным является через связь операции что совпадения номенклатурной группы кода. В свою очередь связи операций, соответствующий идентификатор группы предпочтительнее связь операции по умолчанию. Поиск осуществляется в следующем порядке:

2.  **Код номенклатуры**=**группы** и **связь номенклатуры**=&lt;код группы номенклатуры&gt;
3.  **Код номенклатуры**=**все**

7.  **Configuration**=
8.  **Узел**=&lt;код для узла&gt;
9.  **Site**=

Таким образом операции можно использовать только один раз для каждого маршрута. Если операция выполняется несколько раз в тот же маршрут, все вхождения этой операции будут иметь одинаковые связь операции, и нельзя будет иметь различные свойства (например, запусков) для каждого экземпляра.

## <a name="route-versions"></a>Версии маршрута
Версии маршрута используются для учета возможных вариантов производства продуктов или предоставления большей управляемости производственного процесса. Они определяют, какой маршрут следует использовать при конкретном выпуске продукта или варианта выпущенный продукт создается. Чтобы определить, какой маршрут используется для используемого продукта можно использовать следующие ограничения:

-   Аналитики продукта (размер, цвет, стиль или конфигурации)
-   Количество в производстве
-   Узел производства
-   Дата производства

Когда вы готовите продукт на определенном узле, в определенном количестве или в определенный период, можно назначить с определенной версией маршрута версия маршрута по умолчанию. Тем не менее Обратите внимание, что для заданной выпущенного продукта и данным набором ограничений разрешена, только одного активного маршрута.  

Параметры управления производством можно потребовать, что задать срок действия версии маршрута всегда.

### <a name="approval-of-route-versions"></a>Одобрение версий маршрута

Прежде чем версия маршрута можно использовать в планировании или производственной технологии, должны быть утверждены. При утверждении версии маршрута, можно также утвердить связанного маршрута. Однако следует отметить, что можно утвердить версию маршрута, только в том случае, если соответствующая маршрута также утверждается.

### <a name="activating-the-default-route-version"></a>Активация версии маршрута по умолчанию

При активации версии маршрута, можно назначить его как версия маршрута по умолчанию, который сводное планирование будет использовать, или будет использоваться для создания производственных заказов. Можно иметь только одну активную версию маршрута для данного набора ограничений (например, период, узла или количество). Если версия, что вы пытаетесь активировать конфликтует с версией, уже активно, вы получаете сообщение об ошибке. Чтобы активация неоднозначным, необходимо затем деактивировать конфликтующие версии или изменить ограничения (обычно точки) на версии маршрута.

### <a name="electronic-signatures"></a>Электронные подписи

Если необходимо вести журнал, записей, которые утверждает и не активирует каждой версии маршрута, можно требуются электронные подписи для решения следующих задач. Пользователи, утверждение и активация версии маршрута этого необходимо проверить его личность, используя [электронной подписи](/dynamics365/operations/organization-administration/electronic-signature-overview).

### <a name="product-change-that-uses-case-management"></a>Изменение продукта, который использует управление обращениями

Продукта изменить регистр утверждение и активацию новые или измененные маршруты и версии маршрутов предоставляет простой способ просмотра и обзора ограничения версии маршрута. Можно также утвердить и активировать все маршруты, которые относятся к определенного изменения в одной операции и документирования результатов в случае изменения продукции.

## <a name="maintaining-routes"></a>Поддержка маршрутов
В зависимости от бизнес-требований можно сократить усилия, необходимые для ведения свои определения процесса.

### <a name="making-routes-independent-of-resources"></a>Создание маршрутов независимых ресурсов

Во многих системах операционного ресурса или группы ресурсов, который должен выполнить операцию должен быть указан в маршруте. Однако в Dynamics 365 для операций, можно определить набор требований, которым должен удовлетворять операционный ресурс были применимы для операции. Таким образом конкретный операционный ресурс или группу ресурсов, следует использовать не обязательно должны быть определены до момента Плановая операция. Эта функция особенно полезна, если имеется много работников или оборудование, можно выполнить ту же операцию.  

Например, указать, что операция требует операционный ресурс из **машина** тип, который имеет **Stamping** способность 20 тонн. Механизм планирования будет затем решить эти требования на конкретный операционный ресурс или группу ресурсов, при планировании операции. Поскольку можно просто указать эти требования вместо привязки операции на определенном компьютере, у вас есть более эффективно. Кроме того обслуживание будет проще, если ресурсы перемещаются или добавляются новые ресурсы.  

Дополнительные сведения о различных типах потребностей в ресурсах и их использовании см. в разделе операций потребностей в ресурсах и [ресурсов возможности](resource-capabilities.md).

### <a name="sharing-routes-across-sites"></a>Совместное использование узлам маршрутов

В случае же продукт в более чем один узел производства и шаги, необходимые для производства продукта являются одинаковыми для всех сайтов, часто можно разработать общий маршрут, который будет использоваться во всех рабочих веб-узлов. Чтобы создать общий маршрут, не указать объект в сам маршрут. Тем не менее по-прежнему необходимо создать версии маршрута, которая связывает общий маршрут с продуктом в каждый узел.  

Необходимо также убедиться, что требования к ресурсам для каждой операции в маршруте не относиться к конкретным операциям ресурсов или групп ресурсов, но вместо этого можно выразить в терминах характеристики необходимых ресурсов. Механизм планирования будет могли назначать соответствующие операционных ресурсов, производство планируется на узле. Например если имеются небольшие различия во время выполнения, или время настройки для определенной производственной операции для конкретного узла, можно указать эту информацию, добавив связи дополнительных операций для данного узла.  

Чтобы воспользоваться всеми преимуществами преимущества общей маршрутов, также следует использовать потребление ресурсов на соответствующие спецификации (BOM). При установке флага для потребления ресурса в строке Спецификации склад и местоположение, которое должны быть использованы сырья из определяется, операция планируется для операционного ресурса. Таким образом склад и место не обязательно быть определены до момента Плановая производства. Таким образом можно сделать Спецификации и маршрута независимо от физического местоположения, в которой производится продукт.

### <a name="standard-operation-relations"></a>Отношения стандартные операции

Если ваше предприятие использует стандартизированный операций в течение всего производства, а в случае почти или изменения во время настройки, время выполнения расчета потребления, вычисления, затрат и т. д., можно получить преимущество от создания связей операций для всех операций по умолчанию. В этом случае следует Избегайте создания связи операций, которые относятся к любой маршрут или выпуска продукции.  

Если также потребности в ресурсах с точки зрения навыки и возможности экспресс-выпуска и внести свои маршруты не зависящие от узла, может помочь свести к минимуму общего бизнес-процессов.  

При использовании этого подхода **связей операции** страница становится основной место для обслуживания, время выполнения и другие свойства.

### <a name="resource-specific-process-times"></a>Время выполнения ресурса

Если не указан как часть потребности в ресурсах для операции операционного ресурса или группы ресурсов, соответствующих ресурсов могут работать на разных скоростях. Таким образом может отличаться от времени, необходимое для выполнения операции. Чтобы решить эту проблему, можно использовать **формула** на связь для указания способа расчета времени выполнения операции. Имеются следующие варианты:

-   **Стандартная** — (по умолчанию параметр) использует только поля из связь операции и умножается на количество по заказу за указанное время выполнения вычисления.
-   **Мощность** – вычисление включает **мощности** из операционного ресурса. Таким образом время зависит от ресурса. Значение, указанное в операционный ресурс является мощность в час. Это значение умножается на количество по заказу и **коэффициент** значение из связь операции.
-   **Пакет** — объем пакета вычисляется с помощью данных из связь операции. Число пакетов и, следовательно, время выполнения затем вычисляется на основе количества по заказу.
-   **Пакет ресурсов** — этот параметр обычно такой же, как **партии** параметр. Тем не менее, вычисление включает **партии мощности** из операционного ресурса. Таким образом время зависит от ресурса.
