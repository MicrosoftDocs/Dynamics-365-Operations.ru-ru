# <a name="project-management-and-accounting"></a>Управление и учет по проектам

[!include[banner](../includes/banner.md)]


Функции управления проектом и учета можно использовать в нескольких отраслях, чтобы обеспечить услуги, произвести продукт, или достичь результата.  

Проект — это группа мероприятий, которые предназначены для предоставления услуги, производства продукта или достижения результата. Проекты потребляют ресурсы и создают финансовые результаты в виде доходов или основных средств.

## <a name="projects-across-industries"></a>Проекты в различных отраслях
Функциональность модуля "Управление и учет по проектам" можно использовать в различных отраслях, как показано на следующем рисунке. [![Проекты в различных отраслях](./media/projects-accross-industries.jpg)](./media/projects-accross-industries.jpg) 

В центре обработки вызовов билет можно использовать для описания набора действий, которые необходимы для разрешения звонка. Консалтинговые компании, такие как организации по управлению или техническому консалтингу либо рекламные агентства, называют свои действия проектами. В маркетинге кампания представляет набор работ, который необходимо выполнить. В основанном на проектах производстве производственный заказ включает различную работу, которую необходимо выполнить для производства некоторых готовых товаров. Независимо от того, какое имя используется, эти проекты включают ресурсы, графики и затраты. Функциональность модуля "Управление и учет по проектам" в Microsoft Dynamics 365 for Finance and Operations, Enterprise edition может помочь с планированием, исполнением и анализом этих проектов.

## <a name="project-phases"></a>Фазы проекта
Хотя следующая последовательность операций направлена на внешние проекты или проекты, которые завершены для одного или нескольких клиентов, функциональность также применяется к внутренним проектам, включающим только затраты. 

![3 стадии проекта](./media/3-stages-of-a-project.png) 

Как показано на рисунке выше, управление и учет по проектам можно разделить на три стадии:

1.  Инициация
2.  Выполнение
3.  Анализ

## <a name="initiate-the-project"></a>Инициация проекта
Во время инициации проекта выполняется несколько ключевых процессов. Можно использовать предложение по проекту для предоставления клиенту оценки по труду, расходам и материалам. Можно записать условия выставления накладных, лимиты и соглашения в контракте по проекту. Можно использовать структурную декомпозицию работ (СДР) для планирования и оценки работы. Можно настроить прогнозы и бюджеты для отслеживания хода выполнения проекта. На следующем рисунке показана структура проекта.[![структура проекта](./media/project-structure1.jpg)](./media/project-structure1.jpg)  

### <a name="create-project-quotations"></a>Создание предложения по проекту

На первой стадии продажи предложение по проекту позволяет сделать клиенту неофициальное предложение. Предложение может включать большое количество элементов, например сведения о продуктах или услугах, общую контактную информацию, специальные торговые соглашения и скидки, возможные налоги и сборы.

Также можно выдать гарантийное письмо для проводки предложения по проекту между организацией и клиентом. После создания предложения по проекту можно создать запрос гарантийных писем для клиента и отправить его в банк. После того как банк утвердит запрос, гарантийное письмо выдается клиенту. 

Дополнительные сведения см. в разделе [Предложения по проекту](project-quotations.md).

### <a name="create-project-contracts"></a>Создание контрактов по проекту

При заключении контракта с клиентом или другим источником финансирования с целью выполнения проекта сначала необходимо создать контракт по проекту. Затем во время создания проекта необходимо назначить его соответствующему контракту. Тип проекта, создаваемый для контракта по проекту, определяет метод, используемый для выставления счетов клиентам проекта. Можно изменить контракт по проекту и связанный проект, но невозможно изменить тип проекта. Дополнительные сведения о типах проектов см. в разделе "Создание проектов".

Дополнительные сведения о контрактах по проектам см. в разделе [Контракты по проектам](project-contracts.md).

### <a name="create-work-breakdown-structures"></a>Создание структурной декомпозиции работ

Степень детализации в СДР зависит от уровня точности, который необходим в оценках, и уровня отслеживать отслеживания, необходимого для этих оценок. Проекты, которые имеют очень низкий допуск отклонения от графика или стоимости, обычно требуют более детальной СДР и тщательного отслеживания хода выполнения работ и затрат по СДР. 

Дополнительные сведения см. в разделе [Структурные декомпозиции работ](work-breakdown-structures.md).

### <a name="create-project-forecasts-and-budgets"></a>Создание прогнозов по проекту и бюджетов

Прогнозирование можно использовать, если организация имеет рабочую перспективу и фокусируется на выручке и затратах по конкретным проводкам. Однако, если организация больше фокусируется на финансовых суммах, можно использовать бюджетирование. Каждый способ имеет свои преимущества. Дополнительные сведения см. в разделе [Прогнозы и бюджеты проекта](project-forecasts-budgets.md).

Проводки в проектах "Время" не разносятся в главную книгу. Они включаются в отчеты использования работников. На проект типа "Время" могут быть отнесены только почасовые проводки. ### <a name="assign-workers-categories-and-resources"></a>Назначение работников, категорий и ресурсов

Можно запланировать трудовые ресурсы на основе требований и расписания проекта или на основе навыков и доступности работников. Используя возможности планирования ресурсов, можно развернуть работников организации эффективно и грамотно. Можно быстро найти наиболее квалифицированных работников для работы над проектом. Можно также легко узнать, как оптимально использовать эти ресурсы в ходе проекта. 

Ниже представлено несколько способов использования функциональности планирования ресурсов:

-   Использовать сведения об атрибутах работника, таких как образование, квалификация, наличие сертификатов и опыт участия в проектах, чтобы оценить соответствие работника требованиям проекта.
-   Использовать сведения о календаре и доступности работника, чтобы согласовать график работника и календарь проекта.
-   Просмотреть возможности каждого работника и определить способы их использования. Например, если работник используется не в полной мере, его можно назначить проекту, соответствующему доступности и атрибутам работника.
-   Просматривать сведения о доступности работников, чтобы убедиться в отсутствии конфликтов календаря с их назначениями.
-   Просматривать сведения об использовании работников в представлении сводки (например по подразделениям или по работникам) или в подробном представлении (например, по работникам в подразделении или по недельным данным для каждого работника).
-   Изменять назначения ресурсов для различных периодов времени (например день, неделя или месяц), чтобы оптимизировать способы использования работников.

## <a name="execute-the-project"></a>Выполнение проекта
Во время выполнения проекта члены группы или менеджеры регистрируют работу и понесенные расходы с помощью табелей учета рабочего времени, отчетов по расходам и других бизнес-документов. Менеджеры проектов имеют инструменты, которые позволяют им контролировать потребление бюджетных сумм для проекта. Менеджеры проектов также могут заказывать, выбирать или приобретать материалы для проектов с помощью заказов на покупку и других бизнес-документов. Подготавливаются и утверждаются накладные, чтобы можно было выставить накладные клиентам за выполняемую работу. Наконец, в ходе этого процесса распознается выручка для определения финансов организации.

### <a name="manage-work-breakdown-structures"></a>Управление структурными декомпозициями работ

СДР — это описание работы, которая будет выполнена по проекту. СДР — это иерархия задач. Она представляет не только работу для каждой задачи, но и размер, стоимость и продолжительность задачи. 

Дополнительные сведения см. в разделе [Структурные декомпозиции работ](work-breakdown-structures.md).

### <a name="manage-project-forecasts-and-budgets"></a>Управление прогнозами и бюджетами проектов

Имеются два способа управления проектами: прогнозы проекта и бюджеты проекта. Прогнозирование можно использовать, если организация имеет рабочую перспективу и фокусируется на выручке и затратах по конкретным проводкам. Однако, если организация больше фокусируется на финансовых суммах, можно использовать бюджетирование.

Дополнительные сведения см. в разделе [Прогнозы и бюджеты проекта](project-forecasts-budgets.md).

### <a name="create-production-orders"></a>Создание производственных заказов

Производственный заказ, связанный с проектом, можно связать с заказом на продажу или потребностью в номенклатуре с помощью метода готовой номенклатуры или метода потребляемой номенклатуры. Кроме того, если производственный заказ был создан вручную, то нет ссылки между производственным заказом и заказом на продажу или потребностью в номенклатуре (нет ссылки на заказ). Однако, если производственный заказ был создан автоматически для выполнения заказа на продажу или потребности в номенклатуре, есть ссылка между производственным заказом и заказом на продажу или потребностью в номенклатуре (ссылка на заказ). 

На основе сочетания этих факторов используйте один из следующих методов:

-   **Готовая номенклатура/ссылка на заказ** — связь проекта с заказом на продажу или потребностью в номенклатуре. С помощью этого метода фактические затраты по проекту разносятся, когда выставляется накладная по заказу на продажу или при обновлении отборочной накладной для потребности в номенклатуре. Стоимость разносится в качестве произведенной номенклатуры.
-   **Готовая номенклатура/нет ссылки на заказ** — невозможно выполнить разноску фактических затрат до тех пор, пока цикл производства для номенклатуры не получит статус **Завершено**. Стоимость для готовой номенклатуры разносится как отдельная проводка.
-   **Потребленная номенклатура/ссылка на заказ** — связь проекта с потребностью в номенклатуре. С помощью этого метода можно просматривать фактические затраты по проекту, когда производство имеет статус **Начато** или "Принято". Затраты разносятся как несколько проводок по номенклатуре по сырью для проекта и времени, используемым для производства. При обновлении отборочной накладной для требования по номенклатуре затраты по проекту не разносятся. Можно также указать уровень в иерархии спецификаций, на котором отслеживаются проекты в производстве.
-   ****Потребленная номенклатура/нет ссылки на заказ**** — связь проекта с потребностью в номенклатуре. С помощью этого метода можно просматривать фактические затраты по проекту, когда производство имеет статус **Начато** или "Принято". Затраты разносятся как несколько проводок номенклатуры по сырью для проекта и времени, используемым для производства. Можно также указать уровень в иерархии спецификаций, на котором отслеживаться проекты в производстве.

### <a name="procure-products-and-services"></a>Закупка продуктов и услуг

Покупки и продажи номенклатур являются доминирующими действиями многих фирм, работающих по проектам.

### <a name="process-project-invoices"></a>Обработка накладных по проекту

Тип проекта определяет те процедуры выставления накладных, которые следует использовать. Выставление накладных возможно только для двух типов внешних проектов: "Время и расходы" и "Фиксированная цена". Проекты "Время и расходы" и проекты "Фиксированная цена" всегда присоединяются к контракту по проекту. 

Перед созданием накладной клиента по проекту можно создать предварительную накладную или предложение по накладной. В предложении по накладной фактуры можно выбрать проводки по проекту для включения в накладную по проекту. После этого можно просмотреть сведений накладной до разноски накладной по проекту и отправить ее клиенту или другому источнику финансирования. 


Дополнительные сведения об обработке накладных по проекту см. в разделе [Выставление накладных по проекту](../accounts-payable/project-invoicing.md).

## <a name="analyze-the-project"></a>Анализ проекта
На самом базовом уровне проект используется для группировки проводок, регистрирующих затраты, и для разноски этих затрат в главную книгу. 

Обычно эти проводки являются результатом бизнес-документов, таких как табели учета рабочего времени, отчеты по расходам или проводки по запасам. Жизненный цикл проекта обычно начинается с оценок, прогнозов и бюджетов, которые помогают планировать и прогнозировать работу и финансовое влияние проекта. При анализе проекта можно оценить не только проводки, которые выполнялись во время проекта, но и точность оценок и прогнозов, коэффициенты использования членов проектной группы и общий успех проекта.

### <a name="analyze-cash-flow"></a>Анализ движения денежных средств

Используйте мониторинг движения денежных средств для отслеживания как прогнозируемого, так и фактического движения денежных средств в проекте. Движение денежных средств можно отслеживать в ходе выполнения проекта или просматривать его в завершенном проекте. 

Отслеживая движение денежных средств, можно оценить отдельный проект, использовать отчеты для просмотра нескольких проектов и переносить движение денежных средств в прогнозы движения денежных средств в главной книге.

#### <a name="cash-inflow-forecasting"></a>Прогнозирование поступления денежных средств

На основании настроек можно спрогнозировать движения денежных средств для выбранного проекта. Например, если дата проекта 5 марта 2012 г. и накладная выставляется 31 марта 2012 г., можно прогнозировать срок выполнения и ожидаемую дату оплаты продаж.

-   **Дата проекта:** 5 марта 2012 г.
-   **Дата накладной:** 31 марта 2012 г. Эта дата основывается на частоте накладных. В данном примере частота накладных определяется текущим месяцем. Это означает, что накладные по всем проводкам, разнесенным в марте, выставляются в последний день месяца.
-   **Срок выполнения:** 14 апреля 2012 г. Эта дата определяется условиями платежа, выбранными для данного проекта. В этом примере для условий платежа выбрано значение 14 дней. Поэтому 14 дней добавляются к дате накладной со сроком выполнения 14 апреля 2012 г.
-   **Ожидаемая дата оплаты продаж:** 27 апреля 2012 г. Эта дата рассчитывается путем добавление числа дней в поле **Общее количество буферных дней** на странице **Параметры модуля "Управление и учет по проектам"** к числу дней в поле **Индивидуальный буфер** на странице **Контракты по проектам** и добавления общего значения к числу дней в поле **Срок выполнения**. В этом примере выбрано значение **3** в поле **Общее количество буферных дней** и значение **10** в поле **Индивидуальный буфер**. Поэтому 13 дней добавляются к сроку выполнения с ожидаемой датой платежа 27 апреля 2012 г.

Общее количество буферных дней может заменить индивидуальные буферные дни или быть добавлено к индивидуальным буферным дням.

-   Чтобы использовать общие буферные дни как замену индивидуальных буферных дней, укажите среднее число дней между сроком выполнения и фактической датой платежа для клиентов.
-   Чтобы добавить общие буферные дни к индивидуальным буферным дням, в поле **Общее количество буферных дней** укажите свою оценку числа дней между днем отправки платежа клиентом и днем его получения вашей организацией.

Настройте индивидуальные буферные дни в контракте по проекту. Число дней рассчитывается с учетом как срока оплаты накладной на продажи, так и опыта компании в отношении графика платежей клиента.

#### <a name="actual-cash-inflow"></a>Фактическое поступление денежных средств

Фактическое поступление денежных средств похоже на прогнозирование, за исключением того, что расчеты можно начать с даты первой накладной. Рассмотрим пример:

-   **Дата накладной:** 2 марта 2012 г.
-   **Срок выполнения:** 16 марта 2012 г. Для условий платежа задано значение 14 дней.
-   **Ожидаемая дата оплаты продаж:** 29 марта 2012 г. В расчет включены три общих буферных дня и 10 индивидуальных буферных дней.

#### <a name="cost-forecasting"></a>Прогнозирование затрат

С учетом определенных дней дата оплаты затрат может отличаться от даты проекта. В этом случае дата оплаты затрат рассчитывается прибавлением числа дней из даты проекта к числу дней в условиях оплаты. 

Например, дата проводки по проекту 5 марта 2012 г. и заданы следующие условия оплаты:

-   **Часы:** текущий месяц (**M**)
-   **Расходы:** 14 дней (**D14**)
-   **Номенклатуры:** 30 дней (**D30**)

С учетом этих параметров дата оплаты расходов для каждого типа проводок будет следующей:

-   **Часы:** 31 марта 2012 г., т. е. последний день выбранного месяца.
-   **Расходы:** 19 марта 2012 г., т. е. 14 дней после даты проводки.
-   **Номенклатуры:** 4 апреля 2012 г., т. е. 30 дней после даты проводки.

> [!NOTE] 
> Вместо этого срок выполнения для заказа на покупку учитывает проводку поставщика при создании заказа на покупку по проекту. Срок выполнения не определяется параметрами по умолчанию. 

Дата оплаты расходов рассчитывается без учета буферных дней. Когда проект будет завершен и все операции по расчету затрат и выставлению накладных будут выполнены, и затраты, и продажи будут разнесены на счета прибылей и убытков. 

Когда все продажи и накладные поставщика завершены, можно просмотреть отношения между полями на странице **Движение денежных средств** и полями на странице **Отчеты по проекту**.

| Страница "Движение денежных средств" | Страница "Отчеты по проекту" |
|----------------|-------------------------|
| Поступление денежных средств   | Выручка                 |
| Расход денежных средств  | Общие затраты              |
| Чистое движение денежных средств | Валовая прибыль            |

### <a name="review-costs"></a>Просмотр затрат

Для отслеживания затрат, которые ваша организация несет во время проекта, используйте страницу **Управление затратами**. Сравнивая себестоимость первоначального бюджета проекта с текущими фактическими затратами, можно определить входит ли проект в рамки бюджета, превышает его или использует не весь бюджет. 

> [!NOTE] 
> При использовании страницы **Управление затратами** для просмотра текущего статуса затрат проектов применяйте прогнозные модели, выбранные для первоначального и оставшегося бюджета. Если выбрать другие прогнозные модели при расчете затрат, результаты расчетов будут неточными.

#### <a name="viewing-the-remaining-budgeted-amounts"></a>Просмотр оставшихся бюджетных сумм

Если в качестве метода управления затратами выбран параметр **Остаток бюджета** на странице **Параметры модуля "Управление и учет по проектам"**, на странице **Управление затратами** рассчитываются затраты, которые не были разнесены как фактические или не были помечены как подтвержденные. В частности, суммы на вкладке **Общее** в нижней области страницы **Управление затратами** вычисляются следующими способами:

-   **Фактические затраты** — общая сумма, потраченная на проект для выбранной строки затрат. Сумма фактических затрат рассчитывается на странице **Обновления главной книги**.
-   **Подтвержденные затраты** — дополнительная сумма расходов, которую юридическое лицо обязуется заплатить. Определенные суммы подтвержденных затрат рассчитываются на странице **Подтвержденные затраты**.
-   **Остаток бюджета** — сумма, первоначально добавленная в бюджет, которая все еще доступна для выбранной строки затрат. Сумма остатка бюджета рассчитывается на странице **Предварительный просмотр главной книги**.
-   **Общие затраты** — сумма фактических затрат, подтвержденных затрат и остатка бюджета.

На странице **Управление затратами** на вкладке **Отклонение** вы можете просмотреть сравнение общих ожидаемых затрат и первоначального бюджета. Это сравнение показывает разницу между этими суммами. Таким образом, можно просмотреть несовпадение данных. Суммы отклонений можно подсчитать следующими способами:

-   **Исходный бюджет** — сумма, первоначально добавленная в бюджет для выбранной строки затрат. Сумма исходного бюджета рассчитывается на странице **Предварительный просмотр главной книги**.
-   **Общие затраты** — сумма фактических затрат, подтвержденных затрат и остатка бюджета, указанная на вкладке **Общее**.
-   **Отклонение** — расхождение между фактическими затратами и исходным бюджетом.
-   **Отклонение на основе количества** — общее расхождение между первоначальным прогнозом и итоговым прогнозом. Расхождение можно выразить математически как (Общее количество прогноза) × (Первоначальная средняя цена – общая средняя цена). Этот расчет используется только для часов в проекте.
-   **Отклонение на основе цены** — общее расхождение между первоначальным прогнозом и итоговым прогнозом. Это расхождение можно выразить математически как (Первоначальная цена прогноза) × (Первоначальное количество прогноза – общее количество прогноза). Этот расчет используется только для часов в проекте.

#### <a name="viewing-the-total-budgeted-amounts"></a>Просмотр итоговых бюджетных сумм

Если в качестве метода управления затратами выбран параметр **Общий бюджет** на странице **Параметры модуля "Управление и учет по проектам"**, на странице **Управление затратами** рассчитываются фактические и итоговые затраты по проекту, что помогает определить расхождения между ними. В частности, на странице **Управление затратами** суммы в столбцах в нижней области вкладки **Общее** вычисляются следующими способами:

-   **Итого бюджетные затраты** — общая бюджетная сумма для выбранной строки затрат.
-   **Фактические затраты** — общая сумма затрат по проекту на текущую дату для выбранных строк затрат.
-   **Подтвержденные затраты** — общая сумма, подтвержденная для выбранной строки затрат.
-   **Отклонение** — расхождение между суммой фактических и подтвержденных затрат и общими затратами. Отклонение покажет, нужно ли указать дополнительные затраты для итогового бюджета.

На странице **Управление затратами** на вкладке **Отклонение** можно просмотреть отличия итогового бюджета и исходного бюджета с помощью следующих полей:

-   **Исходный бюджет** — сумма, первоначально добавленная в бюджет для строки затрат. Исходный бюджет рассчитывается на странице **Предварительный просмотр главной книги**.
-   **Итого бюджетные затраты** — итоговые затраты, первоначально добавленные в бюджет для строки затрат. Общие бюджетные затраты рассчитывается на странице **Предварительный просмотр главной книги**.
-   **Отклонение** — отклонение для строки затрат. Эта сумма рассчитывается путем вычитания общих затрат из исходного бюджета.
-   **Отклонение на основе количества** — общее расхождение между исходным бюджетом и общим бюджетом. Чтобы вычислить эту сумму, нужно вычесть общее число часов бюджета из числа часов первоначального бюджета, а затем умножить полученную разность на первоначальную бюджетную цену затрат. Это расхождение можно выразить математически как (Первоначальная бюджетная цена затрат) * (Первоначальные бюджетные часы – общее количество бюджетных часов). Этот расчет используется только для часов в проекте.
-   **Отклонение на основе цены** — чтобы вычислить эту сумму, нужно вычесть общее число часов бюджета из числа часов исходного бюджета, а затем умножить полученную разность на общее потребленное число часов. Это расхождение можно выразить математически как (Общее количество использованных часов) × (Первоначальные бюджетные часы – общее количество бюджетных часов). Этот расчет используется только для часов в проекте.

### <a name="analyze-utilization"></a>Анализ использования

Коэффициент использования — это процент времени в конкретном периоде работы, в течение которого работник занят оплачиваемой или производительной работой. Оплачиваемые часы — это часы рабочего времени работника, за которые определенному клиенту можно выставить накладную. 

Коэффициент использования работника рассчитывается путем деления числа оплачиваемых часов на число рабочих часов в конкретном периоде времени. Например, если у работника число оплачиваемых часов в периоде времени равно 30, а число часов работы в том же периоде времени равно 40, то коэффициента использования работника составляет 75%. 

При расчете коэффициента использования работника можно рассчитать либо коэффициент оплаты, либо коэффициент эффективности.

-   **Коэффициент оплаты** — разность между оплачиваемыми и неоплачиваемыми часами, или нормо-часы.
-   **Коэффициент эффективности** — разность между продуктивными и непродуктивными часами, или нормо-часы. Продуктивные часы — это часы, затраченные работником на работу над конкретным проектом. Счет на оплату продуктивных часов обычно (за исключением внутренних проектов) выставляется клиенту. Счет клиенту на оплату непродуктивных часов никогда не выставляется.

Коэффициент использования рассчитывается на странице **Почасовое использование**. Расчеты основаны на предпочтительных значениях по умолчанию. Эти предпочтительные значения также определяют способ расчета часов путем задания значений **Использование** или **Накладные расходы** для каждого типа проекта. Это применимо к расчету и коэффициентов оплаты, и коэффициентов эффективности.

-   **Использование** — часы, зафиксированные для проекта выбранного типа, всегда считаются оплачиваемым или эффективным использованием.
-   **Накладные расходы** — часы, зафиксированные для проекта выбранного типа, всегда считаются неоплачиваемым или неэффективным использованием.
-   **В соответствии со свойством строки** — свойства строки конкретной почасовой проводки определяют, будут ли часы считаться оплачиваемым или эффективным использованием.
-   **Не включено** — при расчете оплачиваемого или эффективного использования часы не учитываются.

На странице **Почасовое использование** помимо общего процента коэффициента использования для работника или проекта можно просмотреть число часов, включенное в расчет коэффициента использования для каждого из следующих типов часов:

-   **Невключенные часы** — -эти часы не включаются в расчет коэффициента использования часов.
-   **Включенные часы** — эти часы рассчитываются сложением учитываемых при начислении часов и часов по накладным расходам. Эти часы включаются в расчет коэффициента использования.
-   **Часы по накладным расходам** — если вы рассчитываете коэффициент оплаты, эти часы совпадают с неоплачиваемыми часами. При расчете коэффициента эффективности эти часы совпадают с непродуктивными часами.
-   **Учитываемые при начислении часы** — если вы рассчитываете коэффициент оплаты, эти часы совпадают с оплачиваемыми часами. При расчете коэффициента эффективности эти часы совпадают с продуктивными часами.

При расчете коэффициента использования для работника можно использовать нормо-часы или включенные часы. При использовании включенных часов следует убедиться, что работники зафиксировали все свое рабочее время в периодах табеля учета рабочего времени. Это необходимо, поскольку результат расчета выражается в виде процента от указанных часов. При вычислении коэффициента использования часов для проекта, контракта на проект, записи клиента или категории необходимо в расчете использовать включенные часы.

### <a name="review-project-statements"></a>Просмотр отчетов по проекту

Можно создать отчет по проекту для быстрого просмотра хода выполнения проекта. При выполнении отчета по проекту можно указать критерии, используемые для расчета отчета, выбрав нужные параметры на вкладке **Общее** на странице **Отчеты по проекту**. Можно включить или исключить следующую информацию:

-   Типы проектов
-   Типы проводок
-   Дата проекта/дата ГК
-   Данные

После расчета отчета можно просмотреть следующую информацию на различных вкладках на странице **Отчеты по проекту**:

-   **Общее** — общие сведения о базовой структуре прибылей и убытков проекта.
-   **Прибыли и убытки** — информация о начисленном доходе.
-   **НЗП** — информация о сальдо счетов НЗП.
-   **Потребление** — информация о потреблении времени, номенклатурах, расходах и проводках по зарплате.
-   **Накладная** — информация о накладных и промежуточных накладных.
-   **Почасовая ставка** — почасовые ставки для часов, разнесенных на счета прибылей и затрат.
