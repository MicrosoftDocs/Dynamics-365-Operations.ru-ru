# <a name="create-vendor-payments-by-using-a-payment-proposal"></a>Создание платежей поставщика с помощью предложения по оплате

Этот раздел содержит обзор параметров предложения по оплате и несколько примеров, которые показывают, как работают предложения по оплате. Предложения по оплате часто используются для создания оплаты поставщикам, потому что запрос можно использовать для быстрого выбора накладных поставщиков для оплаты на основе таких критериев, как срок оплаты и скидка по оплате. 

Организации часто используют предложения оплаты для создания оплаты поставщикам, потому что запрос предложения оплаты можно использовать для быстрого выбора накладных поставщиков для оплаты на основе срока оплаты, скидки по оплате и других критериев. 

Запрос предложения по оплате содержит различные вкладки, каждая из которых имеет различные параметры для выбора оплачиваемых накладных. **Параметр** вкладка содержит параметры, которые наиболее часто используемых большей части организации. На **записи для включения** экспресс-можно указать, какие накладные или поставщиков, включаемых для платежа, определяя диапазоны для различных характеристик. Например чтобы оплатить определенный диапазон поставщиков, можно определить фильтр для диапазона поставщика. Эта функция часто используется для выбора накладных для конкретного метода оплаты. Например, при определении фильтра которой **способ оплаты** = **проверка**, выбранных только накладные, имеющие этого способа платежа для платежа, условии, что пароль также должен соответствовать других критериях, указанных в запросе. Вкладка **Дополнительные параметры** содержит дополнительные параметры, некоторые из которых могут не относиться к вашей организации. Например, эта вкладка содержит варианты для оплаты накладных с помощью системы централизованных платежей.

## <a name="parameters"></a>Параметры
-   **Выбор накладных по** — накладных в диапазоне дат, указанный в параметре **от даты** и **Дата** поля может быть выбран по дате оплаты, дате скидки при оплате наличными или того и другого. При использовании даты скидки при оплате наличными, система сначала выполняет поиск накладных, имеющих дату скидки при оплате наличными между Начальная дата и конечная дата. Система после этого определяет, подходит ли накладная для скидки по оплате, путем использования даты сеанса для того, чтобы убедиться, что дата скидки по оплате еще не прошла.
-   **Начальная дата** и **Конечная дата** — накладные, которые имеют дату срока оплаты или скидки по оплате внутри этот диапазона дат, выбираются для оплаты.
-   **Дата оплаты** — если дата определена, то все платежи создаются для этой даты. Поле **Минимальная дата платежа** игнорируется.
-   **Минимальная дата платежа** — впишите минимальную дату платежа. Например **от даты** и **Дата** полей укажите диапазон от 1 сентября до 10 сентября, а дата минимальный платеж является 5 сентября. В этом случае все накладные, срок, с 1 сентября 5 сентября имеют дату платежа 5 сентября. Тем не менее все накладные, срок, от 5 сентября 10 сентября иметь дату оплаты, которая совпадает с датой выполнения каждого счета-фактуры.
-   **Предельная сумма** — впишите максимальную полную сумму для всех платежей.
-   **Создание платежей без предварительного просмотра накладной** — Если этот параметр имеет значение **Да**, платежи создается непосредственно на **платежей поставщикам** страницы. **Предложение по оплате от** страница будет пропущена. Поэтому платежи будут созданы более быстро. Платежи можно все еще изменить на странице **Платежи поставщику**. Альтернативно вы можете вернуться на страницу **Предложение по оплате** с помощью кнопки **Изменить накладные для выбранного платежа**.

## <a name="advanced-options"></a>Дополнительные параметры
-   **Проверка сальдо поставщика** — если для этого параметра задано значение **Да**, перед оплатой накладной система проверяет, не ли у поставщика дебетового сальдо. Если поставщик дебетовое сальдо, платеж не создается. Например у поставщика может быть кредит-ноты или платежи, которые были разнесены, но еще не были сопоставлены. В таких случаях платежи поставщику не требуется. Вместо этого кредитные авизо или платежи должны быть сопоставлены неоплаченным накладным.
-   **Удалить отрицательные оплаты** — этот параметр работает по-разному в зависимости от того, сделаны ли оплаты для индивидуальных накладных или для суммы накладных, которые соответствуют критериям оплаты. Это поведение определяется в методе платежа.
-   **Платеж для каждой накладной** — если для параметра **Удалить отрицательные оплаты** задано значение **Да** и несопоставленная накладная и оплата существуют для поставщика, то только накладная выбирается для оплаты. Существующий платеж не сопоставляется с накладной. Если для параметра **Удалить отрицательные оплаты** задано значение **Нет** и накладная и платеж не сопоставлены, для платежа выбираются как накладная, так и платеж. Платеж создается для платежа, и возмещение (отрицательная оплата) создается для платежа.
-   **Оплата для суммы накладных** — если для параметра **Удалить отрицательные оплаты** задано значение **Да** и несопоставленная накладная и оплата существуют для поставщика, то для платежа выбираются как несопоставленная накладная, так и платеж, и суммы складываются, формируя полную сумму платежа. Единственное исключение происходит, если сумма приводит к возмещению. В этом случае не выбираются на накладная, ни платеж. Если ** удалить отрицательные оплаты ** включен режим **нет**и не сопоставление накладной и платежа, выбранные для оплаты накладной и платежа и суммы будут добавлены складываются для получения итоговой суммой платежей.
-   **Только печать отчета** — установите для этого параметра значение **Да** для того, чтобы посмотреть результаты предложения по оплате в отчете, но не создавать никакие платежи.
-   **Включить накладные поставщика от других юридических лиц** — если ваша организация имеет централизованный процесс для оплаты, и предложение оплаты должно включать накладные от других юридических лиц, которые включены в критерии поиска, установите для этого параметра значение **Да**.
-   **Предложить отдельный платеж поставщика для каждого юридического лица** — если для этого параметра установлено значение **Да**, то отдельная оплата создается для каждого юридического лица на каждого поставщика. Поставщик в платеже — это поставщик из накладной от каждого юридического лица. Если для этого параметра задано значение **Нет** и поставщик имеет накладные в нескольких юридических лицах, один платеж создается для общей суммы выбранных накладных. Поставщик в платеже — это поставщик у текущего юридического лица. Если у текущего юридического лица нет счета поставщика, используется счет поставщика первой накладной, подлежащей оплате.
-   **Валюта оплаты** — в этом поле указывается в валюте, которые создаются все платежи в. Если валюта не определена, оплаты каждой накладной в валюте накладной.
-   **День недели оплаты** — введите день недели, когда оплата должна быть сделана. Это поле используется только в том случае, если метод платежа настроен для того, чтобы суммировать накладные для платежа в определенный день недели.
-   **Тип корр. счета** и **корр. счет** — установить эти поля для определения определенного типа счета (например, **главной книги** или **банка**) и корреспондирующий счет (например, конкретный банковский счет). Определяет способ оплаты для накладной, тип корр. счета по умолчанию и корреспондирующий счет, но эти поля можно использовать для переопределения значений по умолчанию.
-   **Дополнительные фильтры** – на **записи для включения** экспресс-можно определить дополнительные диапазоны условий. Например необходимо заплатить только диапазон поставщиков, можно определить фильтр для диапазона поставщика. Эта функция часто используется для выбора накладных для конкретного метода оплаты. Например, при определении фильтра которой **способ оплаты** = **проверка**, выбранных только накладные, имеющие этого способа платежа для платежа, условии, что пароль также должен соответствовать других критериях, указанных в запросе.

## <a name="country-specific-considerations"></a>Сведения, специфичные для конкретных стран
### <a name="norway"></a>Норвегия

#### <a name="dimension-control"></a>Аналитический контроль

Контроль аналитик позволяет контролировать группирование созданных строк по предложениям по оплате и задавать аналитики по умолчанию на основе финансовых аналитик, использованных для примененных накладных. В контексте страны Норвегии для каждого способа оплаты имеется вкладка финансовой аналитики, где можно сделать активным контроль аналитики, а также включить группирование для каждой аналитики. Возможны следующие варианты:

-   Поле **Аналитический контроль** отключено. Предложение по оплате ведет себя так же, как и для любой другой страны.
-   Поле **Аналитический контроль** активировано без дальнейшего определения аналитик. Предложение по оплате создается без учета аналитик. Созданная проводка не наследует никаких аналитик из примененной записи.
-   Поле **Аналитический контроль** активировано и дополнительные аналитики включены. Теперь вы определяете, как аналитики будут копироваться в журнал. Например: • выберите **BusinessUnit** флажок, чтобы создать предложение по оплате по бизнес-единицу для метода оплаты, выберите • **центр затрат** флажок, чтобы создать предложение по оплате по центрам затрат для данного способа оплаты

**Примечание.** Если выбрано несколько аналитик в третьем варианта, предложение по оплате создается для комбинаций аналитик.

#### <a name="bank-account-selection"></a>Выбор банковского счета

Можно определить стандартный счет дебетового платежа для каждого способа платежа независимо от контекста страны. Это задается в строках платежа, созданных предложением. С помощью функции банковского счета можно определить несколько дебетовых банковских счетов, управляемых аналитикой и валютой либо сочетанием этих параметров, для использования различных дебетовых банковских счетов в зависимости от каждой комбинации. Можно настроить эти комбинации в **способов оплаты** страницу с использованием **банковские счета** кнопки доступны для каждого способа оплаты с **тип счета разноски** = **банка**.
