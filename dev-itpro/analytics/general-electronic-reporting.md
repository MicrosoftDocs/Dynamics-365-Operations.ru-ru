# <a name="electronic-reporting-overview"></a>Обзор электронной отчетности

[!include[banner](../includes/banner.md)]


В этой статье представлен обзор средства электронной отчетности (ER). Сюда входят сведения о ключевых понятиях, сценарии, поддерживаемые электронной отчетностью, и список форматов, которые были разработаны и выпущены в рамках решения.

Электронная отчетность (ER) — это средство, которое можно использовать для настройки форматов электронных документов в соответствии с юридическими требованиями различных стран и регионов. Электронная отчетность позволяет управлять этими форматами в течение их жизненного цикла. Например, можно принять новые нормативные требования и создавать бизнес-документы в требуемом формате для электронного обмена информацией с государственными учреждениями, банками и другими сторонами. Механизм электронной отчетности нацелен на бизнес-пользователей, а не на разработчиков. Поскольку вы настраиваете форматы, а не код, процессы создания и корректировки форматов электронных документов проходят быстрее и проще. В настоящее время электронная отчетность поддерживает форматы листа TEXT, XML и OPENXML. Однако интерфейс расширения предоставляет поддержку дополнительных форматов.

## <a name="capabilities"></a>Возможности
Механизм электронной отчетности обладает следующими возможностями:

-   Он представляет собой единый инструмент для электронной отчетности в различных доменах и заменяет более 20 различных механизмов, обеспечивающих ту или иную форму электронной отчетности в Microsoft Dynamics 365 for Operations.
-   Он изолирует формат отчета от текущей реализации Dynamics 365 for Operations. (Другими словами, формат применяется к различным версиям Dynamics 365 for Operations.)
-   Он поддерживает создание пользовательского формата на основании исходного формата. Он включает возможности автоматического обновления настраиваемого формата при внесении изменений в исходный формат, поскольку вводятся требования к локализации и настройке.
-   Он становится основным стандартным средством для поддержки требований локализации в электронной отчетности как для корпорации Майкрософт, так и для ее партнеров.
-   Он поддерживает возможность распределения форматов среди партнеров и клиентов посредством Microsoft Dynamics Lifecycle Services (LCS).

## <a name="concepts"></a>Концепции
### <a name="components"></a>Компоненты

Электронная отчетность поддерживает два типа компонентов: **модель данных** и **формат**.

#### <a name="data-model-components"></a>Компоненты модели данных

Компонент модели данных — это абстрактное представление структуры данных, которое используется для описания определенной области бизнес-домена с детализацией, достаточной для удовлетворения требований к отчетности, предъявляемых к этому домену. Компонент модели данных состоит из следующих частей:

-   Модель данных в виде набора характерных для домена бизнес-объектов и иерархически структурированного определения отношений между ними.
-   Сопоставления модели, которое связывает выбранные источники данных Dynamics 365 for Operations с отдельными элементами этой модели данных во время выполнения, поток данных и заполнение бизнес-данными с компонентом модели данных.

Бизнес-объект модели данных представлен как контейнер (запись). Свойства бизнес-объекта представлены как элементы данных (поля). Каждый элемент данных имеет уникальное имя, метку, описание и значение. Значение каждого элемента данных можно разработать таким образом, чтобы оно распознавалось как строка, целое число, вещественное значение, дата, тип перечисления, логическое значение и т. д. Кроме того, оно может быть другой записью или списком записей. Один компонент модели данных может содержать несколько иерархий бизнес-объектов для конкретного домена и сопоставления модели, поддерживающие характерный для определенного отчета поток данных во время выполнения. Иерархии будут дифференцироваться по одной записи, выбранной в качестве корня сопоставления модели. Например, модель данных области домена платежей может поддерживать следующие сопоставления:

-   Компания -&gt; Поставщик -&gt; Проводки по оплате домена расчетов с поставщиками
-   Клиент -&gt; Компания -&gt; Проводки по оплате домена расчетов с клиентами

Обратите внимание, что бизнес-объекты (например, "Компания" и "Проводки по оплате") разрабатываются только один раз. После этого они повторно используются различными сопоставлениями. Сопоставление обладает следующими возможностями:

-   Оно может использовать различные типы данных Dynamics 365 for Operations как источники данных для модели данных. Например, оно может использовать таблицы, объекты данных, методы или перечисления.
-   Оно поддерживает параметры пользовательского ввода, которые можно определить как источники данных для модели данных, когда какие-либо данные должны быть указаны во время выполнения.
-   Оно поддерживает преобразование данных Dynamics 365 for Operations для получения желаемого группирования, фильтрации, сортировки, суммирования; кроме того, к ним можно добавлять логические вычисляемые поля, разработанные в виде Microsoft Excel-подобных формул (дополнительные сведения см. в разделе [Конструктор формул в электронной отчетности](general-electronic-reporting-formula-designer.md)).

[![Редактор Excel-подобных формул](./media/pic-formula-1024x615.png)](./media/pic-formula.png) Для каждого бизнес-домена разрабатывается компонент модели данных, используемый в качестве унифицированного источника данных для отчетности, изолирующего отчетность от физической реализации источников данных Dynamics 365 for Operations, и представляющий характерные для домена бизнес-концепции и функциональность в форме, которая делает первоначальную разработку и дальнейшую поддержку форматов отчетности более эффективной.

#### <a name="format-components"></a>Компоненты формата

Компонент формата — это схема выходных данных отчетности, которые будут создаваться во время выполнения. Схема состоит из следующих элементов:

-   Формат, который определяет структуру и содержимое документа электронной отчетности, формируемого во время выполнения.
-   Источники данных в виде набора параметров пользовательского ввода и характерной для конкретного домена модели данных, которая использует выбранное сопоставление модели.
-   Сопоставление формата в виде набора привязок источников данных формата к отдельным элементам формата, задающим во время выполнения поток данных и правила формирования выходных данных формата.
-   Проверка формата в виде набора настраиваемых правил, управляющих создание отчетов во время выполнения в зависимости от контекста выполнения (например, правило, которое останавливает создание выходных данных платежей поставщика и выдает исключение, если отсутствуют определенные атрибуты выбранного поставщика, например номер банковского счета).

Компонент формата поддерживает следующие функции:

-   Создание выходных данных отчетности в виде отдельных файлов в различных форматах: текстовом формате, XML или лист.
-   Создание нескольких файлов отдельно и инкапсуляция этих файлов в ZIP-файлы.

Компонент формата предоставляет возможность прилагать определенные файлы, которые могут использоваться в выходных данных отчетности:

-   Книги Excel, содержащие лист, который можно использовать в качестве шаблона для выходных данных в формате листа OPENXML.
-   Другие файлы, которые могут включаться в выходные данные формата в качестве предопределенных файлов.

#### <a name="component-versioning"></a>Управление версиями компонентов

Для компонентов электронной отчетности поддерживается управление версиями. Для управления изменениями в компонентах электронной отчетности предусмотрен следующий workflow-процесс:

-   Версия, которая была создана первоначально, помечается как версия **ЧЕРНОВИК**. Эту версию можно изменять, и она доступна для выполнения тестов.
-   Версию **ЧЕРНОВИК** можно преобразовать в версию **ЗАВЕРШЕНО**. Эту версию можно использовать в локальных процессах отчетности.
-   Версию **ЗAВЕРШЕНO** можно преобразовать в версию **ИСПОЛЬЗУЕТСЯ СОВМЕСТНО**. Эта версия публикуется в LCS и может использоваться в процессах глобальной отчетности.
-   Версию **ИСПОЛЬЗУЕТСЯ СОВМЕСТНО** можно преобразовать в версию **ПОДДЕРЖКА ПРЕКРАЩЕНА**. Эту версию можно удалить.

Версии со статусом **ЗАВЕРШЕНО** или **ИСПОЛЬЗУЕТСЯ СОВМЕСТНО** доступны для других вариантов обмена данными. В компоненте, который имеет эти статусы, можно выполнять следующие действия с ними:

-   Они могут быть сериализованы в формате XML и экспортированы из Dynamics 365 for Operations как файл в формате XML.
-   Они могут повторно сериализованы в XML-файл и импортированы в Dynamics 365 for Operations как новая версия компонента электронной отчетности.

#### <a name="component-date-effectivity"></a>Дата начала действия компонента

Версии компонента электронной отчетности могут иметь дату вступления в силу. Для компонента электронной отчетности можно задать дату **Действует с**, чтобы указать дату, с которой компонент вступает в силу для процессов отчетности. Для определения того, является ли компонент действительным для выполнения, используется дата сеанса Dynamics 365 for Operations. Если на определенную дату действительно несколько версий, для процессов отчетности используется последняя версия.

#### <a name="component-access"></a>Доступ к компонентам

Доступ к компонентам формата электронной отчетности зависит от настроек кода страны/региона по ISO. Если этот параметр не задан для выбранной версии конфигурации формата, доступ к формату компонента можно получить из любой компании Dynamics 365 for Operations во время выполнения. Когда этот параметр содержит коды стран/регионов по ISO, компонент формата доступен только из тех компаний Dynamics 365 for Operations, у которых основной адрес находится в стране/регионе, заданных одним из кодов стран/регионов по ISO компонента формата. У разных версий компонента формата могут быть разные коды стран/регионов по ISO.

#### <a name="configuration"></a>Конфигурация

Конфигурация электронной отчетности представляет собой оболочку для того или иного компонента электронной отчетности: **модель данных** или **формат**. Конфигурация может включать различные версии определенного компонента электронной отчетности. Каждая конфигурация помечается как принадлежащая определенному поставщику конфигураций. Версия **ЧЕРНОВИК** компонента конфигурации доступна для редактирования, когда владелец конфигурации выбран в качестве активного поставщика в настройках электронной отчетности в Dynamics 365 for Operations. Каждая конфигурация модели содержит компонент **Модель данных**. Новая конфигурация формата может происходить (наследоваться) из определенной конфигурации модели данных. Создаваемая конфигурация формата будет представлена в дереве конфигурации как дочерний элемент исходной конфигурации модели данных. Создаваемая конфигурация формата содержит компонент **Формат**. Компонент **Модель данных** исходной конфигурации модели будет автоматически вставлен в компонент **Формат** дочерней конфигурации формата в качестве источника данных по умолчанию. Конфигурация электронной отчетности используется совместно различными компаниями Dynamics 365 for Operations.

#### <a name="provider"></a>Поставщик

Поставщик электронной отчетности — это идентификатор субъекта, используемый для указания автора (владельца) каждой конфигурации электронной отчетности. Электронная отчетность позволяет управлять списком поставщиков конфигурации. Владельцем конфигураций форматов, выпущенных для электронных документов в рамках решения Dynamics 365 for Operations, является поставщик конфигураций **Майкрософт**.

#### <a name="repository"></a>Репозиторий

В репозитории электронной отчетности хранятся конфигурации электронной отчетности. В данный момент поддерживаются следующие типы репозиториев электронной отчетности: **Ресурсы Operations** и **Проект LCS**. Репозиторий **Ресурсы Operations** предоставляет доступ к списку конфигураций, которые выпущены в составе решения Dynamics 365 for Operations корпорацией Майкрософт, выступающей в роли поставщика конфигураций электронной отчетности. Эти конфигурации можно импортировать в текущий экземпляр Dynamics 365 for Operations и использовать для электронной отчетности. Их также можно использовать для дополнительной локализации и настройки. Репозиторий **проект LCS** предоставляет доступ к списку конфигураций определенного проекта LCS (библиотека основных средств проекта LCS), который был выбран на стадии регистрации репозитория. Электронная отчетность позволяет отправлять совместно используемые конфигурации из текущего экземпляра Dynamics 365 for Operations в определенный репозиторий **Проект LCS**. Можно импортировать конфигурации из определенного репозитория **Проект LCS** в текущий экземпляр Dynamics 365 for Operations. Необходимые репозитории **Проект LCS** можно зарегистрировать отдельно для каждого поставщика конфигураций текущего экземпляра Dynamics 365 for Operations. Каждый репозиторий может быть предназначен для определенного поставщика конфигураций.

-   Сделать содержимое более понятным во время разработки для разработчиков форматов, говорящих на других языках, которые будут использовать модель данных для сопоставления данных компонентов формата.

### <a name="storing-a-designed-model-component-as-a-model-configuration"></a>Хранение разработанного компонента модели как конфигурации модели

## <a name="handling-er-components"></a>Обработка компонентов электронной отчетности
### <a name="publishing-an-er-component-in-lcs-to-offer-it-externally-localization"></a>Публикация компонента электронной отчетности в LCS для внешнего использования (локализация)

Владелец созданного компонента (модели или формата) может использовать электронную отчетность для публикации завершенной версии компонента в LCS. Требуется репозиторий типа **проект LCS** для текущего поставщика конфигурации электронной отчетности. Когда статус завершенной версии компонента меняется с **ЗАВЕРШЕНО** на **ИСПОЛЬЗУЕТСЯ СОВМЕСТНО**, эта версия публикуется в LCS. После публикации компонента в LCS владелец этого компонента становится поставщиком службы для поддержки этого компонента. Например, если компонент формата предназначен для создания установленного законом электронного документа (например, в соответствии со сценарием локализации), предполагается, что этот формат будет соответствовать изменениям в законодательстве и что поставщик будет выпускать новые версии компонента всякий раз, когда возникают новые законодательные требования. Чтобы ознакомиться с этим сценарием подробнее, воспроизведите руководство по задаче **Электронная отчетность. Отправка конфигурации в Lifecycle Services** (входит в состав бизнес-процесса **7.5.4.3 Приобретение/разработка компонентов ИТ-услуг и решений (10677)**).

### <a name="importing-an-er-component-from-lcs-to-use-it-internally"></a>Импорт компонента электронной отчетности из LCS для внутреннего использования

Электронная отчетность позволяет импортировать компоненты электронной отчетности из LCS в текущий экземпляр Dynamics 365 for Operations. Требуется репозиторий типа **проект LCS**. После импорта компонента электронной отчетности из LCS в текущий экземпляр Dynamics 365 for Operations владелец этого экземпляра становится потребителем службы, предоставляемой владельцем (автором) импортированного компонента. Например, если компонент формата предназначен для создания определенного электронного документа из Dynamics 365 for Operations в формате для определенной страны/региона (сценарий локализации), предполагается, что потребитель службы сможет получать все обновления этого формата для обеспечения его совместимости с законодательными требованиями. Чтобы ознакомиться с этим сценарием подробнее, воспроизведите руководство по задаче **Электронная отчетность. Импорт конфигурации из Lifecycle Services** (входит в состав бизнес-процесса **7.5.4.3 Приобретение/разработка компонентов ИТ-услуг и решений (10677)**).

### <a name="building-a-format-selecting-another-format-as-a-base-customization"></a>Построение формата путем взятия другого формата за основу (настройка)

Электронная отчетность позволяет создать (наследовать) новый компонент из текущей версии компонента (базового), который был импортирован из LCS. Например, пользователь хочет получить новый формат для реализации специальных требований к электронному документу (например, дополнительное поле или подробное описание) для поддержки сценария настройки. Чтобы ознакомиться с этим сценарием подробнее, воспроизведите руководство по задаче **Электронная отчетность. Обновление формата путем взятия для него новой базовой версии** (входит в состав бизнес-процесса **7.5.4.3 Приобретение/разработка компонентов ИТ-услуг и решений (10677)**).

### <a name="upgrading-a-format-selecting-a-new-version-of-base-format-rebase"></a>Обновление формата путем взятия для него новой версии базового формата (перебазирование)

Электронная отчетность позволяет автоматически внедрять изменения последней версии базового компонента в текущую черновую версию производного компонента. Этот процесс называется *повторным размещением*. Например, новое изменение в нормативной базе, которое было введено в последней версии компонента формата, импортированного из LCS, может быть автоматически внедрено в пользовательскую версию этого формата электронного документа. Любые изменения, которые не удалось внедрить автоматически, считаются конфликтами. Эти конфликты будут представлены для разрешения вручную в средстве конструктора для соответствующего компонента. Чтобы ознакомиться с этим сценарием подробнее, воспроизведите руководство по задаче **Электронная отчетность. Обновление формата путем взятия для него новой базовой версии** (входит в состав бизнес-процесса **7.5.4.3 Приобретение/разработка компонентов ИТ-услуг и решений (10677)**).
