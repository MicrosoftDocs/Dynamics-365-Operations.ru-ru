## <a name="customer-data-flow"></a><span data-ttu-id="e1070-108">Поток данных клиентов</span><span class="sxs-lookup"><span data-stu-id="e1070-108">Customer data flow</span></span>

<span data-ttu-id="e1070-109">*Клиент* — это хорошо определенная концепция в приложениях.</span><span class="sxs-lookup"><span data-stu-id="e1070-109">*Customer* is a well-defined concept in applications.</span></span> <span data-ttu-id="e1070-110">Таким образом, интеграция данных клиентов просто включает в себя согласование концепции клиента между двумя приложениями.</span><span class="sxs-lookup"><span data-stu-id="e1070-110">Therefore, the integration of customer data just involves harmonizing the customer concept between the two applications.</span></span> <span data-ttu-id="e1070-111">На следующем рисунке показан поток данных клиентов.</span><span class="sxs-lookup"><span data-stu-id="e1070-111">The following illustration shows the customer data flow.</span></span>

![Поток данных клиентов](media/dual-write-customer-data-flow.png)

<span data-ttu-id="e1070-113">Клиенты могут быть широко классифицированы на два типа: коммерческие/организационные клиенты и потребители/конечные пользователи.</span><span class="sxs-lookup"><span data-stu-id="e1070-113">Customers can be broadly classified into two types: commercial/organizational customers and consumers/end users.</span></span> <span data-ttu-id="e1070-114">Эти два типа клиентов хранятся и обрабатываются по-разному в Finance and Operations и Common Data Service.</span><span class="sxs-lookup"><span data-stu-id="e1070-114">These two types of customers are stored and handled differently in Finance and Operations and Common Data Service.</span></span>

<span data-ttu-id="e1070-115">В Finance and Operations как коммерческие/организационные клиенты, так и потребители/конечные пользователи обрабатываются в одной таблице, которая называется **CustTable** (CustomerCustomerV3Entity), и классифицируются на основе атрибута **Тип**.</span><span class="sxs-lookup"><span data-stu-id="e1070-115">In Finance and Operations, both commercial/organizational customers and consumers/end users are mastered in a single table that is named **CustTable** (CustomerCustomerV3Entity), and they are classified based on the **Type** attribute.</span></span> <span data-ttu-id="e1070-116">(Если для параметра **Тип** задано значение **Организация**, клиент является коммерческим/организационным клиентом, а если для параметра **Тип** задано значение **Физическое лицо**, то клиент является потребителем/конечным пользователем.) Информация об основном контактном лице обрабатывается через сущность SMMContactPersonEntity.</span><span class="sxs-lookup"><span data-stu-id="e1070-116">(If **Type** is set to **Organization**, the customer is a commercial/organizational customer, and if **Type** is set to **Person**, the customer is a consumer/end user.) The primary contact person information is handled through the SMMContactPersonEntity entity.</span></span>

<span data-ttu-id="e1070-117">В Common Data Service коммерческие/организационные клиенты обрабатываются в сущности "Организация" и идентифицируются в качестве клиентов, когда атрибут **RelationshipType** имеет значение **Клиент**.</span><span class="sxs-lookup"><span data-stu-id="e1070-117">In Common Data Service, commercial/organizational customers are mastered in the Account entity and are identified as customers when the **RelationshipType** attribute is set to **Customer**.</span></span> <span data-ttu-id="e1070-118">Как потребители/конечные пользователи, так и контактное лицо представлены сущностью "Контакт".</span><span class="sxs-lookup"><span data-stu-id="e1070-118">Both consumers/end users and the contact person are represented by the Contact entity.</span></span> <span data-ttu-id="e1070-119">Чтобы обеспечить четкое разделение между потребителем/конечным пользователем и контактным лицом, сущность **Контакт** имеет флаг типа Boolean, который называется **Для продажи**.</span><span class="sxs-lookup"><span data-stu-id="e1070-119">To provide a clear separation between a consumer/end user and a contact person, the **Contact** entity has a Boolean flag that is named **Sellable**.</span></span> <span data-ttu-id="e1070-120">Когда параметр **Для продажи** имеет значение **True**, контакт является потребителем/конечным пользователем, и для этого контакта могут быть созданы предложения с расценками и заказы.</span><span class="sxs-lookup"><span data-stu-id="e1070-120">When **Sellable** is **True**, the contact is a consumer/end user, and quotations and orders can be created for that contact.</span></span> <span data-ttu-id="e1070-121">Когда параметр **Для продажи** имеет значение **False**, контакт является просто основным контактным лицом клиента.</span><span class="sxs-lookup"><span data-stu-id="e1070-121">When **Sellable** is **False**, the contact is just a primary contact person of a customer.</span></span>

<span data-ttu-id="e1070-122">Когда контакт, для которого невозможны продажи, участвует в процессе предложений с расценками или заказа, для параметра **Для продажи** устанавливается значение **True**, чтобы пометить контакт как контакт для продажи.</span><span class="sxs-lookup"><span data-stu-id="e1070-122">When a non-sellable contact participates in a quotation or order process, **Sellable** is set to **True** to flag the contact as a sellable contact.</span></span> <span data-ttu-id="e1070-123">Контакт, который стал контактом для продажи, остается контактом для продажи.</span><span class="sxs-lookup"><span data-stu-id="e1070-123">A contact that has become a sellable contact remains a sellable contact.</span></span>
