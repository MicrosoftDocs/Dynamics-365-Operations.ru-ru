# <a name="store-inventory-management"></a>Управление запасами магазина

[!include [banner](includes/banner.md)]

При работе с запасами в Dynamics 365 for Retail и использовании POS-приложения обратите внимание, что POS предоставляет ограниченную поддержку для складских аналитик и определенных типов складируемых номенклатур.

Решение POS не поддерживает следующие конфигурации номенклатур:

- Номенклатуры спецификаций (за исключением наборов продуктов, которые используют некоторые компоненты платформы спецификаций)
- Номенклатуры, учитываемые в двух единицах измерения
- Номенклатуры, контролируемые по партиям

POS-приложение в настоящее время не поддерживает следующие аналитики отслеживания в POS:

- Аналитика отслеживания партии
- Аналитика ответственного

Решение POS предоставляет ограниченную поддержку для следующих аналитик. Ограниченная поддержка указывает, что POS может автоматически устанавливать эти аналитики по умолчанию в складские проводки на основе конфигурации настройки склада или магазина. POS не будет полностью поддерживает аналитики таким образом, как они поддерживаются при ручном вводе проводки продаж в ERP. 

- **Местонахождение склада** — пользователи не будут иметь возможности управлять местоположением склада приемки для номенклатур, полученных на склад магазина, когда магазин не был настроен на использование процесса управления складом. Для этих номенклатур будет использоваться местоположение приемки по умолчанию, определенное на складе магазина. Если для магазина активирован процесс управления складом, включается ограниченная поддержка, которая предлагает пользователю выбрать место приемки для всего прихода. Номенклатуры, проданные из магазина, всегда будут продаваться из местоположения по умолчанию продажи в розницу, определенного в настройках склада магазина. Местоположение для управления запасами возврата можно контролировать с помощью определения местоположения возврата по умолчанию на складе магазина или на основе кодов причин возврата, определенных в политике местоположения возврата.
- **Грузоместо** — грузоместа применяются только в том случае, если параметр **Использовать процессы управления складом** был включен в номенклатуре и складе магазина. В POS, если запасы получены на склад магазина, в котором процесс управление складом включен, и местоположение, выбранное для получения номенклатуры, привязано к профилю местоположения, которому требуется управление грузоместом, приложение POS будет систематически применять грузоместо к строке приемки. Пользователи в POS не смогут изменять эти данные грузоместа, а также управлять ими. Если требуется полное управление грузоместами, рекомендуется, чтобы магазин использовал мобильное WMS или серверный клиент ERP для управления приемкой этих номенклатур.
- **Серийный номер** — приложение POS имеет ограничение на поддержку одного серийного номера для регистрации в строке продаж проводок для заказов, созданных на POS-терминале с серийными номенклатурами. Этот серийный номер не проверяется по зарегистрированным серийным номерам, которые уже имеются в запасах. Если заказ на продажу создается в канале центра обработки вызовов или выполняется через ERP-сервер и несколько серийных номеров регистрируются в одну строку продаж во время процесса выполнения в ERP, эти серийные номера не могут быть применены или проверены, если возврат обрабатывается в POS для этих заказов.
- **Статус запасов** — для номенклатур, использующих процесс управления складом и для которых требуется статус запасов, это поле статуса не может быть установлено или изменено в приложении POS-терминала. Статус запасов по умолчанию, определенный в конфигурации склада магазина, будет использоваться при получении номенклатур в запасы.

> [!NOTE]
> Все организации должны проверить конфигурации номенклатур через разрабатываемый POS-терминал или в среде тестирования перед их развертыванием в производственных средах. Выполните тестирование номенклатур, выполнив обычные денежные проводки продажи без доставки и создав заказы клиента (если применимо) через POS с вашими номенклатурами. Тестирование должно включать выполнение полных процессов разноски отчетов в среде тестирования и проверки того, что проблем не существует.
>
> Настройка номенклатур таким способом, который не поддерживается в POS-приложении, без правильного тестирования может привести к ошибкам процесса разноски журнала операций в производственной среде без простого способа устранения проблем. При необходимости можно рассматривать настройки от партнеров или заказчиков для приложения, чтобы обеспечить успешное выполнение этих процессов разноски. Если настройки не требуются, организации необходимо убедиться, что конфигурация продуктов для ваших продуктов выполнена таким образом, который поддерживается стандартным процессам POS-приложения/создания заказа/разнесения отчета.

## <a name="purchase-orders"></a>Заказы на покупку

Заказы на покупку создаются в головном офисе. Если розничный склад включен в заголовок заказа на покупку, то заказ можно получить в магазине путем использования Modern POS (MPOS) или Cloud POS в Microsoft Dynamics 365 for Retail через операцию **Комплектация/получение**. После ввода количеств, полученных в магазине, в поле **Немедленное получение** в POS для документа заказа на покупку их можно сохранить локально или отправить. Сохранение этих данных локально не влияет на складские запасы. Сохранение должно выполняться только в том случае, если пользователь не готов к разноске прихода в головной офис и ему просто нужен способ временного сохранения введенных данных **Немедленное получение**. В этом случае данные немедленного получения сохраняются локально в базе данных канала пользователя. После обработки документа с помощью параметра **Подтвердить** данные **Немедленное получение** направляются в головной офис и приемка заказа на покупку будет разнесена. 

## <a name="transfer-orders"></a>Заказы на перемещение

Заказ на перемещение может указывать на то, что конкретный магазин является местоположением, из которого могут быть отгружены номенклатуры, или местоположением, в которое могут быть приняты запасы. Если пользователь POS-терминала является складом отгрузки для заказа на перемещение, он сможет ввести количество **Отгрузить сейчас** из POS-терминала. Данные, введенные магазином отгрузки, могут быть сохранены локально или отправлены. При локальном сохранении обновление документа заказа на перемещение в головном офисе не производится. Сохранение должно выполняться только в том случае, если пользователь не готов к разноске отгрузки в головной офис и ему просто нужен способ временного сохранения введенных данных **Отгрузить сейчас**. После того как магазин будет готов к подтверждению отгрузки, следует выбрать параметр **Подтвердить**. При этом отгрузка по заказу на перемещение будет разноситься в головном офисе, чтобы получающийся склад мог получать относительно нее. 

Если пользователь POS-терминала является складом приемки для заказа на перемещение, он сможет ввести количество **Немедленное получение** из POS-терминала. Данные, введенные магазином получения, могут быть сохранены локально или отправлены. Сохранение должно выполняться только в том случае, если пользователь не готов к разноске прихода в головной офис и ему нужен способ временного сохранения введенных данных **Немедленное получение**. В этом случае данные немедленного получения сохраняются локально в базе данных канала пользователя. После обработки документа с помощью параметра **Подтвердить** данные **Немедленное получение** направляются в головной офис и приемка заказа на перемещение будет разнесена. Важно отметить, что получающий магазин будет ограничен только возможностью подтвердит количества приема, которые равны или меньше отгруженных количеств. Попытка получить количества по заказу на перемещение, которые еще не были ранее отгружены, приведет к ошибкам, и приемка не будет подтвержден в головном офисе.

## <a name="stock-counts"></a>Инвентаризация запасов

Инвентаризация запасов может быть плановой или незапланированной. Плановая инвентаризация инициируется головным офисом, при этом головной офис определяет, для каких номенклатур следует провести инвентаризацию. Головной офис создает документ инвентаризации, который можно получить в магазине и в котором указывается фактическое количество запасов в наличии в MPOS или Cloud POS. Внеплановая инвентаризация инициируется магазином, количества фактически доступных запасов обновляются в MPOS или Cloud POS. В отличие от плановых инвентаризаций во время внеплановой инвентаризации не используется предопределенный список номенклатур. По завершении инвентаризации любого типа соответствующая документацию утверждается и отправляется в головной офис. В головном офисе выполняется проверка и разнесение инвентаризации в отдельных шагах.

## <a name="inventory-lookup"></a>Поиск запасов

Текущее количество продукта в наличии в нескольких магазинах и складах можно просмотреть на странице **Поиск запасов**. Помимо текущего количества в наличии, для каждого отдельного магазина можно просмотреть будущие доступные для заказа (ATP) количества. Чтобы это сделать, выберите магазин, для которого вы хотите просмотреть ATP, и щелкните **Показать доступность магазина**.
