# <a name="set-up-fraud-alerts"></a>Настройка оповещений о мошенничестве

[!INCLUDE [banner](includes/banner.md)]

В этом разделе объясняется, как настроить критерии и правила для блокировки потенциально мошеннических заказов на продажу для дополнительного исследования. Функция проверки на мошенничество используется для определения действительности сведений в заказе на продажу. Если сведения в заказе на продажу выглядят недостоверными на основе принятых в организации критериев и правил обнаружения мошенничества, заказ может быть заблокирован для дополнительной проверки администратором.

> [!NOTE]
> Эта функция может использоваться только с обработкой заказа на продажу в канале центра обработки вызовов Retail. 

Если канал центра обработки вызовов определен, параметр **Включить заполнение заказа** должен иметь значение **Да**. Если завершение заказа включено, пользователи могут просмотреть краткие сведения по заказу и щелкнуть **Отправить** для завершения заказа. Пользователям могут также быть предоставлены возможности вручную блокировать заказы на продажу для проверки на предмет мошенничества. Заказы на продажу, введенные пользователем центра обработки вызовов, обрабатываются в рамках правил и критериев проверки на мошенничество во время процесса отправки.

Существует два типа критериев мошенничества, которые используются системой для проверки, требуется ли заблокировать заказ для проверки на мошенничество:

-   **Статические правила** используют конкретное значение, например номер телефона, который был внесен в черный список или адрес электронной почты, помеченный как использовавшийся для прошлых мошеннических операций. На странице **Статические данные о мошенничестве** сведения о мошенничестве могут быть добавлены вручную или с помощью импорта данных вместе с оценками, добавленными к сведениям о мошенничестве. Если включена проверка на мошенничество, каждый введенный заказ на продажу будет сравниваться со статическими данными. Если данные находятся в адресе клиента для выставления счетов или адресе поставки, или если данные находятся в адресе доставки в строке продаж, оценки будут суммироваться для всех уникальных совпадений.  
-   **Динамические правила** состоят из переменных и условий, определенных для этих переменных. С помощью динамических правил можно проверять другие критерии, не включенные в статические правила. Более сложные операторы "И/ИЛИ" могут использоваться для учета несколько условий, чтобы определить, имеется ли положительное соответствие условиям правила в отправляемом заказе на продажу. Например, если пользователь хочет блокировать для проверки на мошенничество все заказов клиентов, связанных к конкретным значением группы клиентов, которые заказывают определенный продукт, условия для проверки клиента и проверка продуктов будет определяться на странице **Правила** с условием типа "И". Заказ блокируется с целью проверки на мошенничество только в том случае, если оба условия выполнены и если значение оценки, назначенное правилу, делает общую оценку вероятности мошенничества для заказа больше минимальной оценки вероятности мошенничества, определенной в параметрах центра обработки вызовов.

Пользователь центра вызовов может также вручную поместить заказ в очередь блокировки для проверки на мошенничество. Если пользователь, который вводит заказ, считает, что клиент, размещающий заказ, может быть подозрительным, и хочет, чтобы какой-то другой пользователь дополнительно проверил действительность заказа перед его обработкой, пользователь, который вводит заказ, может выбрать параметр **Удержание вручную по мошенничеству** в раскрывающемся меню **Удержания** на странице **Сводка по заказу на продажу** (отображается после вызова функции заказа **Завершено**). Пользователю будет предложено ввести заметку для дополнительного пояснения, почему он считает, что заказ может быть мошенническим, чтобы проверяющий имел дополнительный контекст.

Все заказы, заблокированные с помощью ручной блокировки мошеннических заказов или в результате вычисления оценки вероятности мошенничества системой, отображаются на странице **Удержания заказов**, на которой заказ может быть проверен и либо отменен, либо одобрен для обработки.

> [!NOTE]
> Использование нескольких правил или слишком сложных правил приведет к низкой производительности системы при отправке заказов на продажу. Функция оповещения о мошенничестве не были оптимизированы для обработки большого количества статических записей данных о мошенничестве и большого числа активных правил. Важно помнить, что каждое правило оценивается при выполнении функции отправки записи заказа на продажу в центре вызова. Правила оцениваются по заголовку заказа на продажу и всем строкам заказа. Чем больше правил и чем сложнее инструкции правил, тем больше времени потребуется для их обработки. Если имеется большое количество строка номенклатуры в заказе, а также большое количество активных правил и статических записей данных, этот систематический процесс просмотра и утверждения всех данных и расчета оценки мошенничества может иметь очень значительное влияние на быстродействие.  Организации, использующие этот компонент, должны всегда тестировать и проверять, что время обработки отправки заказа является приемлемым, прежде чем применять какие-либо изменения в правилах или статических критериях выявления мошенничества в рабочей среде.
