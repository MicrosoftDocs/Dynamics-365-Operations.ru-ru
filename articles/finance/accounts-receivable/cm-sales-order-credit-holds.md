# <a name="credit-holds-for-sales-orders"></a>Удержания кредита по заказам на продажу
[!include [banner](../includes/banner.md)]

В этой теме описывается настройка правил, используемых для помещения заказа на продажу в блокировку кредита. Правила блокировки управления кредитом могут применяться к отдельному клиенту или к группе клиентов. Правила блокировки определяют реакцию на следующие обстоятельства:

1. Число просроченных дней
2. Статус счетов
3. Условия оплаты
4. Срок кредитного лимита истек
5. Сумма просрочки
6. Сумма по заказу на продажу
7. Часть используемого доступного кредита

Кроме того, имеется два параметра, которые управляют дополнительными сценариями, которые будут блокировать заказ на продажу

1. Изменение условий оплаты
2. Изменение скидок при сверке

## <a name="set-up-blocking-rules-and-exclusion-rules"></a>Настройка правил блокировки и правил исключений

Когда клиент инициировал проводку по продажам, информация в заказе на продажу анализируется на основе набора правил блокировки, которые позволяют решить, следует ли расширять кредит для клиента и разрешать ли перемещение продажи вперед. Можно также определить исключения, которые будут переопределять правила блокировки и разрешать обработку заказа на продажу. Можно настроить правила блокировки и правила исключения на странице **Управление кредитом > Настройка > Настройка управления кредитом >Правила блокирования**.

### <a name="days-overdue"></a>Просрочка (в днях)

Откройте вкладку **Просроченные дни**, если правило блокировки применяется для клиента с одной или несколькими накладными, срок которых просрочен на определенное число дней.
1. Выбор диапазона клиентов, для которого это правило **Допустимый для**.
   - Выберите **Таблица**, если правило применяется к конкретному клиенту.
   - Выберите **Группа**, если правило применяется на уровне группы клиентов. 
   - Выберите **Все**, если правило применяется ко всем клиентам.
2. После указания диапазона необходимо указать **Счет/группа**, которые будут использоваться в диапазоне.
   - Для диапазона **Таблица** при поиске будет представлен список клиентов, которых необходимо выбрать. 
   - Выберите **Группа**, если правило применяется к кредитной группе клиентов.
   - Выберите **Все**, если правило применяется ко всем клиентам. 
3. Выберите **Группа риска**, чтобы использовать критерии для применения блокировки управления кредитованием для клиентов, сгруппированных по общему набору факторов, таких как рейтинг Dun and Bradstreet, число лет, в течение которых они в бизнесе, количество времени, в течение которого они ваш клиент и т. д.  
4. Выбор типа настраиваемого правила. Параметр **Блокировка** создаст правило, блокирующее заказ. Параметр **Исключение** создаст правило, которое будет исключать другое правило из блокировки заказа. 
5. Выбор **Тип значения**. Запись по умолчанию является фиксированным количеством дней. При создании исключения можно вместо этого указать фиксированное число дней или сумму. 
6. Введите количество дней **Просрочено**, которое будет разрешено для выбранного правила блокировки до того, как заказ будет помещен в анализ блокировки управления кредитованием. Число дней просрочки представляет собой дополнительное число льготных дней, добавляемых к числу дней за пределами даты оплаты, до того, как накладная может считаться просроченной. Если указан **Тип значения** как сумма для исключения, введите сумму и валюту для этой суммы.

### <a name="accounts-status"></a>Статус счетов

Откройте вкладку **Статус счета**, если правило блокировки применяется к клиенту с выбранным статусом счета.
1. Выбор типа настраиваемого правила.  **Блокировка** создаст правило, блокирующее заказ. **Исключение** создает правило, которое будет исключать правило из блокировки заказа. 
2. Выбор **Статус счета**, который будет причиной того, что правило помещает заказ на продажу в блокировку или исключает его.

### <a name="terms-of-payment"></a>Условия оплаты

Выбор **Условия оплаты**, если правило блокировки применяется к выбранному условию оплаты.
1. Выбор типа настраиваемого правила.  **Блокировка** создаст правило, блокирующее заказ. **Исключение** создает правило, которое будет исключать правило из блокировки заказа. 
2. Выбор **Условия оплаты**, что будет причиной того, что правило помещает заказ на продажу в блокировку или исключает его.

### <a name="credit-limit-expired"></a>Срок кредитного лимита истек

Откройте вкладку **Окончание срока действия кредитного лимита**, если правило блокирования применяется к клиентам с истекшими кредитными лимитами.
1. Выбор диапазона клиентов, для которого это правило **Допустимый для**.
   - Выберите **Таблица**, если правило применяется к конкретному клиенту.
   - Выберите **Группа**, если правило применяется на уровне группы клиентов. 
   - Выберите **Все**, если правило применяется ко всем клиентам.
2. После указания диапазона необходимо указать **Счет/группа**, которые будут использоваться в диапазоне.
   - Для диапазона **Таблица** при поиске будет представлен список клиентов, которых необходимо выбрать. 
   - Выберите **Группа**, если правило применяется к управлению кредитной группы клиентов.
   - Выберите **Все**, если правило применяется ко всем клиентам. 
3. Выбор **Группа риска** для дальнейшего ограничения списка клиентов, которые будут помещены в блокировку управления кредитованием. 
4. Выбор типа настраиваемого правила. 
   - Выберите **Блокировка**, чтобы создать правило, блокирующее заказ. 
   - Выберите **Исключение**, чтобы создать правило, которое будет исключать другое правило из блокировки заказа. 
5. Введите **Число дней окончания срока действия кредитного лимита** для выбранного правила блокировки до того, как заказ будет помещен в блокировку управления кредитованием. Число дней просрочки представляет дополнительные льготные дни, добавляемые к количеству дней, на которое просрочен кредитный лимит.

6. Выберите **Тип значения**, который определяет тип значения, которое будет использоваться для проверки того, сколько было использовано кредитного лимита. Правила блокировки требуют наличия процентного значения, но исключение может иметь фиксированную сумму или процент. Пороговое значение связано с кредитным лимитом.
7. Введите **Пороговое значение кредитного лимита** для выбранного правила до того, как клиент перейдет в блокировку управления кредитованием. Это может быть сумма или процент на основе типа значения, выбранного в поле типа значения.

## <a name="put-a-sales-order-on-hold-based-on-other-criteria"></a>Перевод заказа на покупку в блокировку на основе других критериев
  
### <a name="rank-payment-terms"></a>Ранжировать условия оплаты  

При изменении условий оплаты можно принудительно выполнять правил управления кредитом. Необходимо ранжировать условия оплаты и назначить им значение ранжирования. При изменении условий оплаты в заказе на условия оплаты с приоритетом над старыми условиями оплаты, тогда заказ будет отправлен в управление кредитом и потребует утверждения.

Ранжирование условий оплаты можно настроить на странице **Управление кредитом > Настройка > Настройка управления кредитом > Ранжирование условий оплаты**.

1. Выберите условия оплаты, поле **Условия оплаты** для ранжирования; при выборе условия описание отображается в поле **Описание**.
2. Выберите ранжирование условия в поле **Ранжирование**. Все значения зависят друг от друга, что позволяет использовать 1, 2, 3 или 10, 20, 30. Можно также использовать одно и то же значение для большинства условий оплаты, чтобы только одно или два условия оплаты вызывали проверку кредита.

### <a name="rank-settlement-discounts"></a>Ранжировать скидки за досрочную оплату   

При изменении скидок при сверке можно принудительно выполнять правила управления кредитом. Необходимо ранжировать скидки при сверке и назначить им значение ранжирования. При изменении скидки при сверке в заказе на скидки при сверке с приоритетом над старыми скидками при сверке, тогда заказ будет отправлен в управление кредитом и потребует утверждения.

Ранжирование условий оплаты можно настроить на странице **Управление кредитом > Настройка > Настройка управления кредитом > Ранжирование скидок при сверке**.

1. Выберите **Скидка по оплате**, которую необходимо ранжировать. Будет отображено **Описание** скидки по сверке.
2. Выберите значение **Ранжирование**. Все значения зависят друг от друга, что позволяет использовать 1, 2, 3 или 10, 20, 30. Можно также использовать одно и то же значение для большинства скидок при сверке, чтобы только одна или две скидки при сверке вызывали проверку кредита.

## <a name="sequence-the-application-of-rules"></a>Последовательность применения правил

Правила выполняются в конкретном порядке, который изменяется в соответствии с потребностями организации. 

- Один экземпляр правил исключения заказов на продажу позволяет переопределить все правила, которые могут блокировать заказ на продажу. Создайте правило исключения для заказа на продажу и отметьте параметр **Запуск в производство заказа на продажу**. Заказ не будет помещен в блокировку, если это правило исключения истинно, и никакие другие правила не будут проверены.
- Правила блокировки могут помещать заказ в блокировку.
- Правила исключения выполняются после правил блокировки. Правила исключения влияют только на правило, для которого они определены. 
- Правила блокировки и исключения выполняются в таблице, затем в группе, а затем в порядке "Все". Из-за этого порядка обработки можно использовать правило блокировки на уровне "Все", которое не будет выполняться, так как выполняется правило исключения на уровне таблицы или группы.
- Исключения не переопределяют правило блокировки, если они находятся на одном уровне. Например, правило исключения на уровне группы не будет переопределять правило блокировки на уровне группы. Нет необходимости настраивать исключения на уровне "Все" за исключением случаев, оговоренных выше, с одним экземпляром правила исключения заказа на продажу. 

Поведение правила **Используемый кредитный лимит** изменится на основе настроек для параметра **Проверить кредитный лимит заказов по продажам**, который находится в форме параметров кредита и сборов.
- Если для параметра установлено значение "Нет", правило использованного кредитного лимита не будет выполняться.
- Если для параметра установлено значение "Да", а для **Сообщение при превышении кредитного лимита** установлено предупреждение, то при превышении кредитного лимита будет выдано предупреждение. Будут выполнены правила **Использованный кредитный лимит**, чтобы проверить наличие правил, которые необходимо выполнить. Однако в этом сценарии правила обычно не добавляются.
- Если для параметра установлено значение "Да", а **Сообщение при превышении кредитного лимита** задано как ошибка, то кредитный лимит будет проверяться, и заказ будет помещен в блокировку, если кредитный лимит превышен. Кроме того, будут выполнены правила **Использованный кредитный лимит**, чтобы проверить наличие дополнительных правил, которые необходимо выполнить. Сообщение об ошибке не будет отображено, но будет отображена причина блокировки **Кредитный лимит превышен**. 

## <a name="settings-that-will-change-the-way-an-order-is-placed-on-hold"></a>Настройки, которые изменяют способ размещения заказа в блокировку

Заказы могут быть исключены из управления кредитом, даже если есть правила. 

- Если изменить настройки **Исключить клиента из управления кредитом** в **Все клиенты > Выбрать клиента > Экспресс-вкладка кредитования и сборов** как **Да**, то заказы для этого клиента обработаны не будут
- Если изменить значение **Исключить из управления кредитом** в **заголовке заказов на продажу** на экспресс-вкладке **Управление кредитом** как **Да**, правила управления кредитом не будут обработаны. Этот параметр может выполняться только клерком по кредитам или менеджером по кредитам.

## <a name="processing-orders-on-hold-using-the-credit-management-hold-list"></a>Обработка заказов в блокировке с использованием списка блокировки управления кредитованием

Список блокировки управления кредита позволяет менеджерам по кредитам просматривать все заказы на продажу, которые были помещены в блокировку, и позволяет удалять блокировки после решения проблем с кредитами. На странице **Список блокировки управления кредита** отображаются все заказы на продажу, которые были помещены в блокировку. Можно просмотреть список блокировки на странице **Все блокировки кредита** (**Управление кредитом > Список блокировки управления кредитом > Все блокировки кредита**).
Заказы на продажу из всех юридических лиц отправляются в один и тот же список блокировки управления кредитом и дают централизованный просмотр всех проводок, требующих внимания. Пользователи увидят только те сведения по юридическим лицам, к которым у них есть доступ.

Заказ на продажу можно разместить в списке блокировки по следующим причинам:
1. У клиента есть накладная, которая просрочена на указанное число дней. 
2. У заказа есть конкретный статус счета.
3. Для заказа заданы особые условия оплаты.
4. Кредитный лимит клиента просрочен.
5. У клиента есть просроченная сумма, и клиент использовал заданный процент своего кредитного лимита
6. Заказ на продажу превышает определенную сумму.
7. Клиент превысил определенный процент своего кредитного лимита.
8. Условия оплаты отличаются от условий оплаты по умолчанию для клиента.
9. Скидки при сверке отличаются от скидки при сверке по умолчанию для клиента.

Причина блокировки отображается для каждого заказа на продажу в списке блокировки. Если имеется более одной причины для блокировки, причина будет отображаться как **Несколько**. Можно использовать меню **Причины блокировки** на панели операций, чтобы просмотреть все причины, по которым заказ на продажу был заблокирован. Можно также просмотреть **Причины блокировки** в информационном поле.

### <a name="releasing-orders-from-the-hold-list-for-processing"></a>Выпуск заказов из списка блокировки для обработки

После изучения причин блокировки и их устранения можно выпустить заказы на продажу для дальнейшей обработки.

1) Выберите строку в списке блокировки. Можно выпустить несколько заказов, выбрав более одной строки.
2) Выберите **Причина запуска в производство** для заказа, который был выбран для запуска в производство.  
3) Введите **Дата проверки** для каждого заказа, который был выбран для запуска в производство.  
4) Выберите меню **Запуск в производство** на панели операций, чтобы выпустить заказ. Это меню будет доступно только после выбора проводок. Пользователю предлагается две возможности:
   - Выберите **С разноской** для удаления блокировки и разноски документа с использованием того же процесса разноски, который был выполнен, когда заказ на продажу был помещен в блокировку. Например, если подтверждение заказа на продажу было помещено в блокировку, подтверждение заказа на продажу будет завершено после выпуска. Будет открыта форма разноски заказа на продажу, позволяющая пользователю разнести подтверждение.
   - Выберите **Без разноски**, чтобы отменить блокировку без выполнения дальнейшей обработки. Теперь заказ на продажу можно разнести вручную.

### <a name="rejecting-orders-in-the-hold-list"></a>Отклонение заказов в списке блокировки
Можно использовать меню **Отклонить** на панели операций, чтобы отклонить заказ на продажу
1. Выберите строку в списке блокировки. Можно выпустить несколько заказов, выбрав более одной строки.
2. Заказ будет удален из списка блокировки, и заголовок заказа на продажу будет обновлен, чтобы показать, что заказ отклонен.

### <a name="automatically-releasing-credit-management-holds"></a>Автоматическое снятие блокировок управления кредитом
Заказы на продажу помещаются в список блокировки на основе правил блокировки. Однако некоторые причины блокировки могут измениться со временем, если заказ на продажу остается в списке блокировки в течение некоторого периода времени. Например, клиент может оплатить свой счет, освободив свой кредитный лимит. 

Можно использовать меню **Оценка для запуска в производство** для просмотра заказов на продажу в списке блокировки и автоматического их запуска в производство в случае устранения причины блокировки.
1.  Выберите меню **Оценка для запуска в производство**
2.  Выберите **Обработать правила блокировки**, чтобы просмотреть все заказы на продажу. Выберите **Обработать правила блокировки для выбранных строк**, чтобы просмотреть только выбранные строки.
3. Появится ползунок, с которым можно выбрать одного клиента. Оставьте раскрывающийся список клиентов пустым для всех клиентов. 
4. После нажатия кнопки "ОК" обработка выполняется в фоновом режиме, и можно продолжать работу над другими задачами. Если выбрать пакетную обработку до того, как будет нажата кнопка "ОК", обработка будет выполняться в пакетном режиме после нажатия "ОК". Обработка заказов в списке блокировки может занять некоторое время, поэтому используйте "Обновить" для обновления статуса заказов. 
5.  Если причина блокировки больше не применима для заказа, причина блокировки считается более недействительной, и при просмотре причин блокировки не будет отображаться галочка рядом с причиной блокировки.
6.  Если все причины блокировки сняты, то новая причина, **Готово к запуску в производство**, добавляется в список причин блокировки. Теперь заказ на продажу можно разнести автоматически.
7.  Если параметр **Автоматически запускать в производство** на вкладке **Кредит и сбор задолженностей > Настройка > Параметры кредита и сбора задолженности > Кредит > Автоматический запуск в производство** задан как **С разноской**, будет выведен запрос на разноску с формой разноски для заблокированного документа.
8.  Если параметр **Автоматически запускать в производство** на вкладке **Кредит и сбор задолженностей > Настройка > Параметры кредита и сбора задолженности > Кредит > Автоматический запуск в производство** задан как **Без разноски**, необходимо выполнить разноску заказа вручную.

### <a name="credit-management-approval-workflow"></a>Workflow-процесс утверждения управления кредитом

Кроме того, можно создать **Workflow- процессы управления кредитом**, чтобы управлять запуском в производство кредитных блокировок. После настройки workflow-процесса с помощью страницы **Управление кредитом > Настройка > Workflow-процессы управления кредитом** заказы, помеченные для запуска или отклонения, будут отправлены в workflow-процесс, где они должны быть сначала утверждены, прежде чем они будут запущены или отклонены. 

Если включить задачи для запуска в производство с разноской или для выпуска без разноски в workflow-процессе, утверждение workflow-процесса также приведет к запуску заказа на продажу. Однако в случае сбоя обработки выпуска вследствие отсутствия информации настройки или других причин необходимо отозвать заказ на продажу из workflow-процесса, устранить проблему, вызвавшую сбой, а затем снова отправить заказ в workflow-процесс.

Если не включить задачи для запуска в производство с разноской или для выпуска без разноски в workflow-процесс, процесс утверждения workflow-процесс просто позволит выпустить заказ на продажу вручную после завершения утверждения.

### <a name="forced-credit-hold"></a>Принудительная блокировка кредита  
  
Время от времени может потребоваться заблокировать заказы на продажу, даже если заказ не соответствует критериям правил блокировки. Например, менеджер по кредиту может получать уведомление о проблеме, не связанной с кредитом, для клиента, а также принять решение вручную блокировать заказы сразу после устранения проблемы. Можно вручную перевести заказ на продажу в блокировку, если такая ситуация происходит.

1. Откройте заказ на продажу, который необходимо поместить в блокировку.  
2. Выберите **Принудительная блокировка кредита** на вкладке **Управление кредитами** в области действий **Управление кредитом**.
3. Выберите **Причина принудительной блокировки**.
4. Нажмите **ОК**. Заказ на продажу будет возвращен в список блокировки управления кредитом.

Можно также принудительно передать несколько заказов в блокировку на странице **Управление кредитом > Периодические задачи > Принудительная блокировка кредита**. Можно, например, поместить все заказы на продажу в блокировку для отдельного клиента.
1. Выберите **Причина принудительной блокировки**. 
2. Щелкните **Записи для добавления**, чтобы выбрать заказы на продажу, которые необходимо заблокировать. 
3. Чтобы обработать выбранные заказы на продажу, щелкните **ОК**.

Заказы на продажу, которые были принудительно переданы в блокировку, не могут быть обработаны workflow-процессом.

#### <a name="releasing-orders-that-were-added-to-the-credit-management-hold-list-with-a-forced-credit-hold"></a>Запуск заказов, добавленных в список блокировки управления кредитом с принудительной блокировкой кредита
Заказы на продажу с причиной принудительной блокировки не могут быть автоматически запущены. Если заказ на продажу был принудительно заблокирован и вы использовали процесс, автоматически выпускающий заказы на продажу, заказ на продажу будет отображаться как **Готово к запуску в производство** и останется в списке блокировки. Чтобы выпустить заказ, необходимо использовать меню **Запуск в производство**.
 
## <a name="free-text-invoices-orders-and-project-invoice-support-in-credit-management"></a>Накладные с произвольным текстом, заказы и поддержка накладных по проекту в управлении кредитами 
Управление кредитованием может быть использовано в настоящее время только для заказов на продажу. Накладные с произвольным текстом, заказы POS и заказы центров обработки вызовов будут использовать временные кредитные лимиты и страховку/гарантии, добавляемые для корректировки кредитного лимита. Они не будут использовать правила блокировки, и они не будут помещаться в список блокировки, если имеется проблема с кредитным лимитом.

Накладные по проекту в управлении кредитами не поддерживаются.
