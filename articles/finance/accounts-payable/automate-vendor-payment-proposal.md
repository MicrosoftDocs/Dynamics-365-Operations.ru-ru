# <a name="automate-vendor-payment-proposals"></a>Автоматизация предложений по оплате поставщикам

[!include [banner](../includes/banner.md)]

Организации, которые платят поставщикам по регулярному расписанию, теперь могут автоматизировать процесс создания предложений по оплате поставщикам. Автоматизация предложений по оплате поставщикам определяет следующие сведения:

- Когда выполняются предложения по оплате
- Какие критерии используются для выбора накладных, которые должны быть оплачены
- В какой журнал платежей поставщикам сохраняются результирующие платежи

Автоматизация предложений по оплате не разносит платежи автоматически. Таким образом, можно продолжать использовать любые процессы проверки и бизнес-правил, которые в настоящее время используются для утверждения созданных платежей.

> [!NOTE]
> Для использования этих функций в управлении функциями должна быть включена функция **Автоматизация предложений по оплате поставщикам**. 

## <a name="define-the-occurrence-of-vendor-payment-proposals"></a>Определение появления предложений по оплате поставщикам

Автоматизация предложений по оплате поставщикам использует структуру автоматизации процессов. Эта структура используется в других бизнес-процессах для определения повторения выбранного процесса. Для предложений по оплате поставщикам доступ к автоматизации можно получить в разделе **Расчеты с поставщиками \> Настройка платежей \> Автоматизация процесса**.

Сначала воспользуйтесь параметром автоматизации **Создать процесс** и выберите **Предложение по оплате поставщикам**. Мастер поможет выполнить процесс настройки предложения по оплате поставщикам.

## <a name="general-page"></a>Общая страница

На странице **Общие** мастера введите название создаваемого предложения по оплате поставщику. Например, если оплата всем внутренним поставщикам осуществляется с помощью чека в понедельник, введите описательное имя, например, **Домашний\_Чек**. Введенное имя отображается в еженедельном представлении процесса автоматизации в рабочей области **Платежи поставщикам**.

Затем задайте владельца созданного журнала платежей. Владелец — это обычно служащий платежа расчетов с поставщиками (AP), который отвечает за журнал платежей после его создания.

Остальные параметры на странице являются общими и используются для определения шаблона событий для данной версии предложения по оплате поставщику. Например, если событие для платежей по чеку выполняется в понедельник, можно определить его таким образом, чтобы оно выполнялось еженедельно, и можно выбрать понедельник как день недели при запуске. Можно также ввести время раннего планирования, например 2:00, чтобы автоматизация процесса выполнялась до начала следующего рабочего дня.

Важно понимать, что вы используете мастер для определения времени обработки предложения по оплате поставщику. Вы не определяете, когда платежи поставщику созданы, распечатаны и разнесены. В представлении по неделям автоматизация процесса для предложений по оплате поставщикам будет отображаться в дни, выбранные в шаблоне возникновения.

Дополнительные сведения о других полях на странице **Общие** см. в документации по автоматизации процесса.

## <a name="vendor-payment-proposal-page"></a>Страница предложений по оплате поставщикам

Следующая страница мастера — это страница **Предложения по оплате поставщикам**. Она используется для определения критериев выбора накладных поставщиков, которые должны быть оплачены. В общем случае те же параметры находятся в предложении по оплате в журнале платежей поставщикам. Однако есть несколько исключений. Например, все даты на этой странице должны определяться как относительные даты, поскольку дата предложения по оплате изменяется каждый раз при выполнении предложения.

### <a name="journal-name"></a>Наименование журнала

В поле **Наименование журнала** определяется наименование журнала, в котором созданы платежи поставщикам. Результаты автоматизации предложения по оплате поставщикам создадут платежи в заданном журнале, но разноска журнала не выполняется.

### <a name="from-date-and-to-date"></a>Начальная дата и конечная дата

Вместо того, чтобы определять начальную и конечную дату для выбора накладных на основе срока выполнения или даты скидки по оплате, необходимо использовать параметр **Определить начальную дату** и поле **Число дней корректировки для конечной даты**, чтобы определить конечную дату. В автоматизации предложений по оплате отсутствует концепция начальной даты.

По умолчанию для параметра **Определить начальную дату** установлено значение **Нет**. При использовании этого значения по умолчанию при выполнении процесса будут выбраны все накладные для платежа, так как не была определена конечная дата.

Если для параметра **Определить начальную дату** установить значение **Да**, используйте поле **Число дней корректировки для конечной даты** для определения даты, когда накладные выбраны в виде указанного количества дней до или после даты выполнения процесса. Это число может быть положительным, отрицательным или нулевым (0). Затем система выполнит оплату накладных, в которых даты оплаты или даты скидки представляют собой указанное количество дней до или после даты выполнения процесса. Например, для всех накладных, которые были оплачены в пятницу или раньше, серия платежей создает платежи всем поставщикам, выполнив проверку в среду. В этом случае установите для поля **Число дней корректировки для конечной даты** значение **2**. Когда возникновение предложения по оплате выполняется в среду, 25 марта, все накладные со сроком выполнения или датой скидки по оплате 27 марта или раньше будут выбраны для оплаты.

### <a name="minimum-payment-date"></a>Минимальная дата платежа

Минимальной датой платежа является самая ранняя дата, используемая при создании платежей. Необходимо сначала установить для параметра **Определить критерий минимальной даты платежа** значение **Да**. Этот параметр позволяет использовать функцию минимальной даты платежа. Если для параметра установить значение **Да**, используйте поле **Число дней корректировки для минимальной даты платежа** для определения даты минимального платежа в виде указанного количества дней до или после даты выполнения процесса. Это число может быть положительным, отрицательным или нулевым (0). Например, серия платежей создает платежи в среду, чтобы включить все платежи с минимальной датой платежа в предшествующий понедельник. В этом случае установите для поля **Число дней корректировки для минимальной даты платежа** значение **-2**.

В этом примере показано, как должны совместно работать поля для начальной даты и минимальной даты платежа. Автоматизация предложений по платежам настраивается для работы в среду. Поле **Число дней корректировки для конечной даты** задается значение **1**, чтобы определить начальную дату на основе срока выполнения. Для поля **Число дней корректировки для минимальной даты платежа** задано значение **-2**. Если автоматизация процесса платежа начинается в среду 25 марта, все накладные, которые были запланированы на 26 марта или раньше, будут включены в предложение по оплате. Предложения по оплате будут создаваться следующим образом:

- Все накладные, которые должны быть оплачены 23 марта или раньше, будут иметь дату платежа 23 марта.
- Накладные, которые должны быть оплачены 24 марта, будут иметь дату платежа 24 марта.
- Накладные, которые должны быть оплачены 25 марта, будут иметь дату платежа 25 марта.
- Накладные, которые должны быть оплачены 26 марта, будут иметь дату платежа 26 марта.

### <a name="summarized-payment-date"></a>Дата сводного платежа

Дата сводного платежа используется, только когда в поле **Период** в способе оплаты накладных задано значение **Итого**. Если в поле **Период** задано значение **Итого** для ваших способов оплаты, необходимо установить для параметра **Определить критерий даты сводного платежа** значение **Да**. Если для параметра установить значение **Да**, используйте поле **Число дней корректировки для даты сводного платежа** для определения даты сводного платежа в виде указанного количества дней до или после даты выполнения процесса. Это число может быть положительным, отрицательным или нулевым (0). Например, серия создает платежи в среду и компания хочет создать сводный платеж на среду. В этом случае установите для поля **Число дней корректировки для даты сводного платежа** значение **0**.

### <a name="records-to-include"></a>Записи для добавления

Параметры фильтра можно все равно определить для предложения по оплате. Если фильтр определен, критерии фильтра не отображаются на странице мастера. Однако их можно просмотреть, вновь открыв фильтр.

Оставшиеся поля для предложения работают так же, как они работают в предложении по оплате в журнале платежей поставщикам. Сведения о других полях см. в разделе [Создание платежей поставщику с помощью предложения по оплате](create-vendor-payments-payment-proposal.md).

> [!NOTE]
> Некоторые поля, связанные со страной/регионом, и некоторые поля государственного сектора еще не доступны в автоматизации предложений по платежам поставщикам.

В зависимости от потребностей мы рекомендуем оценить, будет ли автоматизация полезна вашей организации.

## <a name="view-the-results-of-a-vendor-payment-proposal-automation"></a>Просмотр результатов автоматизации предложения по оплате поставщикам

После создания серии автоматизации предложения по оплате поставщику ситуации для каждого платежа отображаются в еженедельном представлении автоматизации процесса. Для платежей поставщику еженедельное представление автоматизация процесса добавлено в рабочее пространство **Платежи поставщикам** и на страницу **Автоматизация процессов**.

[![Еженедельное представление автоматизации процесса в рабочей области платежей поставщикам.](./media/vendor-payment-proposal-1.png)](./media/vendor-payment-proposal-1.png)

В еженедельном представлении автоматизации процесса в рабочей области **Платежи поставщикам** отображается только автоматизация предложений по оплате поставщикам. В нем отображаются все платежи для текущей недели для всех юридических лиц, для которых у вошедшего пользователя имеются разрешения безопасности. Например, если служащий по платежам AP сотрудник отвечает за платежи в компаниях USMF и USSI, он увидит ситуации автоматизации предложения по оплате поставщикам для этих двух компаний, но не для других компаний.

[![Еженедельное представление автоматизации процессов для компаний USMF и USSI.](./media/vendor-payment-proposal-2.png)](./media/vendor-payment-proposal-2.png)

Каждое событие отображает компанию, в которой был или будет создан журнал платежей. Если платежи создаются с помощью централизованных платежей, показывается компания, в которой будут создаваться платежи. Это не обязательно означает, какие накладные компаний будут оплачиваться.

Для идентификации предложения по оплате также отображается наименование каждого события.

Кроме того, отображается состояние каждого события. Используются следующие статусы:

- **Запланировано** — предложение по оплате запланировано, но еще не запущено.
- **Ошибка** — было выполнено предложение по оплате, но произошла ошибка. Ошибки можно просмотреть, нажав кнопку **Просмотреть результаты**.
- **Завершено** — предложение по оплате успешно выполнено. Платежи можно просмотреть, нажав ссылку **Просмотреть платежи**. По ссылке открывается журнал платежей, который был создан для событием.

После создания платежей можно просмотреть суммы платежей в журнале. Все суммы показаны в валютах проводки. Например, если журнал платежей содержит платежи как в долларах США, так и в канадских долларах, отображаются итоговые платежи для каждой валюты. 

Журнал платежей можно удалить после того, как он будет создан в ходе автоматизации процессов. Если платеж полностью удален, происходят следующие события:

- Статус автоматизации процесса для данной недели остается **завершенным**.
- Процесс удаляет итоговые суммы платежа, а ссылка **Просмотреть платежи** заменяется кнопкой **Просмотреть результаты**.
- При просмотре результатов выводится сообщение о том, что исходный журнал был удален.

После удаления платежа накладные снова будут открыты для платежа. После этого можно создать новое событие для повторной оплаты накладных.

## <a name="edit-a-vendor-payment-proposal-automation"></a>Изменение автоматизации предложений по оплате поставщикам

Структура автоматизации процессов позволяет редактировать платеж, серию и события, созданные для предложения по оплате. Эту серию можно редактировать на странице **Автоматизация процесса** или в еженедельном представлении автоматизации процесса. Например, если менеджер AP решил создать все чеки для внутренних поставщиков в среду вместо понедельника, он может найти событие в еженедельном представлении и выбрать **Просмотр/правка — серии**. При редактировании серии система предложит указать, следует ли вносить изменения во все существующие события или только в новые события. Исторические события, которые уже имеют статус **завершено** или завершены со статусом **ошибка**, не изменятся.

Можно также добавить новое или изменить существующее событие. Например, выполнение следующего события предложения по оплате запланировано на среду 1 января, но эта дата — праздник. Событие можно изменить на странице **Автоматизация процесса** или в еженедельном представлении автоматизации процесса. Открывается страница, в которой отображаются сведения о расписании и критерии предложения по оплате. Здесь можно изменить запланированное время и дату. Если необходимо внести изменения, можно также изменить критерии предложения по оплате. Например, если изменить запланированную дату события платежа с 1 января на 2 января, можно также изменить относительные даты для конечной даты.

Кроме того, можно отключить событие или серию. Чтобы отключить событие и приостановить обработку, выберите его в еженедельном представлении автоматизации процесса, а затем выберите **Отключить**. Можно отключить серию на странице **Автоматизация процесса**.

## <a name="security-for-payment-proposal-automations"></a>Защита автоматизации предложений по оплате

Для автоматизации предложений по оплате поставщикам были добавлены следующие обязанности и привилегии. Эти обязанности и привилегии являются параметрами безопасности по умолчанию, но их можно изменить в соответствии с требованиями организации. Например, если не только менеджер AP, но и сотрудник по платежам AP могут редактировать и создавать повторы расписания, назначьте обязанность **Вести повторы расписания** для сотрудника с ролью сотрудника по платежам AP.

| Пошлина                              | Роль                                                                       | Привилегии |
|-----------------------------------|----------------------------------------------------------------------------|------------|
| Ведение серий расписания          | Менеджер по расчету с поставщиками                                                   | Эта обязанность предоставляет права на создание и ведение серий автоматизации предложений по оплате и событий с учетом следующих привилегий:<ul><li>Ведение событий расписания</li><li>Ведение серий расписания</li><li>ProcessScheduleOccurrenceListMaintain</li><li>Просмотр еженедельного представления событий</li></ul> |
| Запрос событий расписания | Сотрудник по платежам расчетов с поставщиками, сотрудник по централизованным платежам расчетов с поставщиками | Эта обязанность предоставляет права на просмотр событий автоматизации предложений по оплате с учетом следующих привилегий:<ul><li>Просмотр событий расписания</li><li>Просмотр еженедельного представления событий</li></ul> |
