# <a name="catch-weight-product-processing-with-warehouse-management"></a>Обработка учета продуктов в двух единицах измерения с помощью управления складом

[!include [banner](../includes/banner.md)]


## <a name="feature-exposure"></a>Действие функции

Для использования системы управления складом для обработки учета в двух единицах измерения продуктов необходимо использовать лицензионный ключ конфигурации, чтобы включить эту функцию. Перейдите в раздел **Администрирование системы \> Настройка \> Конфигурация лицензии**. Затем на вкладке **Конфигурационные ключи** разверните **Торговля \> Управление складом и транспортировкой** и установите флажок **Учет в двух единицах измерения для склада**.

> [!NOTE]
> Как конфигурационный ключ лицензии **Управление складом и транспортировкой**, так и конфигурационный ключ лицензии **Распределение процессов \> Учет в двух единицах измерения** также должны быть включены. Для настройки конфигурационных ключей для учета в двух единицах измерения необходимо также включить функцию с помощью рабочей области **Управление функциями**. Основной функцией, которая должна быть включена, является **Обработка учета продуктов в двух единицах измерения с помощью управления складом**. Две связанные, но необязательные функции, которые могут потребоваться включить, это **Изменения состояния запасов для продуктов учетом в двух единицах измерения** и **Использование существующих маркеров учета в двух единицах измерения при приемке производственных заказов**.

После включения лицензионного ключа при создании выпущенного продукта можно выбрать **Учет в двух единицах измерения**. Выпущенный продукт можно также связать с группой аналитик хранения, для которой выбран параметр **Использовать процессы управления складом**.

Перед использованием продукта в управлении складом необходимо выполнить некоторые базовые настройки конкретного продукта для учета в двух единицах измерения.

- Определите номинальное преобразование веса между вторичной единицей учета (коробка) и единицей складского учета (килограмм \[кг\]) как часть определения преобразования единиц измерения.
- Определите минимальный и максимальный допуск веса как часть настройки учета в двух единицах измерения.
- Настройте группу упорядочения единиц измерения, где вторичная единица измерения определяется как низшая единица складского учета (SKU).
- Настройте политику обработки номенклатур, учитываемых в двух единицах измерения.

Дополнительные сведения см. в разделе [Настройка и ведение номенклатур с учетом в двух единицах измерения](https://docs.microsoft.com/dynamicsax-2012/appuser-itpro/setting-up-and-maintaining-catch-weight-items).

## <a name="transaction-adjustments"></a>Корректировки проводок

Так как вес запасов при поступлении на склад может отличаться от веса при выдаче запасов со склада, при обработке продуктов с учетом в двух единицах измерения необходимо корректировать имеющиеся запасы.

> [!NOTE]
> Действие мобильного устройства приведет к корректировке проводок только в том случае, если для параметра отклонения исходящего веса при политике обработки номенклатуры с учетом в двух единицах измерения задано **Разрешить отклонение веса**.

**Пример 1**

Во время производственного процесса **Приемка** входящий вес грузоместа, содержащего восемь коробок продукта с учетом в двух единицах измерения регистрируется как 80,1 кг. Грузоместо затем хранится в области готовой продукции и в течение периода хранения часть веса будет потеряна в воздухе.

Позже, в качестве части процесса комплектации заказа на продажу, вес этого же грузоместа регистрируется как 79,8 кг. Таким образом, в системе появится разница веса как часть набора физической аналитики.

В этом случае система автоматически корректирует разницу путем разноски проводки для отсутствующих 0,3 кг.

**Пример 2**

В определении продукта он настроен для допуска минимального веса 8 кг и максимального веса 12 кг для единицы **Коробка** учета в двух единицах измерения.

Имеется две коробки продукта, и у них есть зарегистрированный вес 16 кг. Если работник склада выбирает и взвесит одну из коробок и вес окажется равным 9 кг, остальная коробка будут весить 7 кг. Однако потому, что 7 кг меньше минимального веса, система выполняет автоматическую коррекцию для повышения веса складских запасов в наличии на 1 кг.

Чтобы настроить счета, на которые разносятся эти корректировки, выберите **Управление затратами \> Настройка политик интеграции главной книги \> Разноска**. Затем на вкладке **Запасы** определите следующие счета:

- Счет убытков во вторичной единице измерения
- Счет прибылей во вторичной единице измерения

## <a name="catch-weight-item-handling-policy"></a>Политика обработки номенклатур, учитываемых в двух единицах измерения

Политика обработки номенклатур с учетом в двух единицах измерения определяет два основных потока управления складом для номенклатур: время регистрации веса номенклатур и порядок его регистрации.

Можно определить, когда вес регистрируется для обработки заказов на продажу и на перемещение. Вес может быть зафиксирован во время любого из следующих процессов:

- **Комплектация** — вес регистрируется во время строк первоначальной работы по подбору для обработки заказа.
- **Упаковка** — вес регистрируется во время упаковки вручную. (Номенклатуры необходимо отправить на станцию упаковки.)

Если фактический вес регистрируется на станции упаковки во время процесса упаковки контейнера, работникам склада не предлагается зарегистрировать вес во время работ по комплектации. Вместо этого средний вес физических запасов используется как вес скомплектованных запасов, которые отправляются в зону упаковки. Эта концепция также применяется к номенклатурам с учетом в двух единицах измерения, отслеживаемым по тегам. Для элементов, отслеживаемых по тегам, эти параметры определяют, когда тег захвачен. Тег может быть записан как во время комплектации с помощью мобильного устройства, так и при ручной упаковке.

> [!NOTE]
> Поскольку при выборе параметра **Упаковка** запасы обновляются в соответствии с выбранным средним весом, это может привести к несоответствию, которое может вызвать корректировку прибылей с учетом в двух единицах измерения и убытков и/или к разнице между весом запасов в наличии и весом по тегу с учетом в двух единицах измерения.

Для внутренних складских процессов управления, таких как инвентаризации и корректировка исправлений, можно определить, требуется ли фиксировать вес. Если не фиксируется, номинальный вес будет использоваться. Другие параметры позволяют записать вес на единицу учета в двух единицах измерения и на количество для расчета.

Можно также определить, как вес регистрируется. В одном из двух основных потоков метки учета в двух единицах измерения отслеживаются и используются для регистрации веса. В другом потоке метки веса учета в двух единицах измерения не отслеживаются.

- **Да** — номенклатура использует метки учета в двух единицах измерения. Каждое число метки указывает одну единицу учета в двух единицах измерения (коробка), и вес и другие сведения связываются с меткой. Для исходящих процессов используется вес, который связан с меткой.
- **Нет** — номенклатура не использует метки учета в двух единицах измерения. Входящие и исходящие процессы взвешивания основаны на фактическом весе, зарегистрированном во время каждого процесса.

Процесс отслеживания меток учета в двух единицах измерения можно использовать для номенклатур, вес которых не изменяется в течение периода хранения. Вес будут записан только во входящем процессе склада. Во время исходящего процесса метки учета в двух единицах измерения будут просто сканироваться, и значения вес, которые связаны с тегами, будут использоваться для обработки исходящих транзакций.

Еще одним важным параметром, связанным с обработкой тегов с учетом в двух единицах измерения, является **Метод отслеживания аналитики тегов учета в двух единицах измерения**. Теги могут быть либо частично отслеживаемыми, либо полностью отслеживаемыми. Если тег частично отслеживается, он отслеживает аналитики продуктов, аналитики отслеживания и состояние запасов. Если тег полностью отслеживается, он отслеживает аналитики продуктов, аналитики отслеживания и **все** аналитики хранения.

Кроме того, при отслеживании тега для номенклатуры имеется параметр **Способ исходящей регистрации веса**. Этот параметр можно настроить таким образом, чтобы в исходящих проводках с мобильного устройства всегда выводился запрос тега. В качестве альтернативы можно настроить параметр таким образом, чтобы при необходимости в нем задавались только нужные теги. Например, имеется пять тегов учета в двух единицах измерения в запасах для указанного грузоместа, и вы указали, что хотите выбрать все пять тегов грузоместа. В этом случае, если параметр **Способ исходящей регистрации веса** задан как **Только запрос тега при необходимости**, пять тегов комплектуются автоматически. Вам не придется сканировать каждый тег. Если для параметра выбрано значение **Всегда запрашивать тег**, необходимо сканировать каждый тег, даже если комплектация включает все пять тегов.

> [!NOTE]
> Как правило, теги захватываются и обновляются только из пунктов меню мобильного устройства. Тем не менее, есть несколько сценариев, в которых теги захватываются в любом месте (например, с помощью ручной станции упаковки). Однако в общем случае пункты меню мобильного устройства должны использоваться для всех складских действий, если используются теги.

### <a name="how-to-capture-catch-weight"></a>Как регистрировать учет в двух единицах измерения

**При использовании отслеживания меток учета в двух единицах измерения** тег должен всегда создаваться для каждой единицы учета в двух единицах измерения, и каждый тег всегда должен быть связан с весом.

Например, **Коробка** представляет собой единицу измерения учета в двух единицах измерения, и вы получаете одну палету с восемью коробками. В этом случае восемь уникальных тегов учета в двух единицах измерения должны быть созданы, и вес должен быть связан с каждым тегом. В зависимости от маркера входящего учета в двух единицах измерения вес всех восьми коробок может быть зафиксирован и средний вес можно распределить на каждую коробку, или уникальный вес может быть зафиксирована для каждой коробки.
При использовании функции **Использование существующих маркеров учета в двух единицах измерения при приемке производственных заказов** с процессом, включенным в пункте меню мобильного устройства, запасы обновляются на основе существующей информации маркера учета в двух единицах измерения. В результате приложение склада не запрашивает запись данных маркера учета в двух единицах измерения в рамках операции приемки готовой продукции.

**При использовании отслеживания маркера учета в двух единицах измерения** вес может регистрироваться для каждого набора аналитик (например, для каждого грузоместа и аналитики отслеживания). Альтернативно вес может быть захвачен на основании общего уровня, такого как пять грузомест (палет).

Для методов для захвата исходящего веса параметр **По вторичной единице измерения** позволяет указать, что взвешивания должны быть выполнены для каждой единицы с учетом в двух единицах измерения (например, для коробки). Параметр **Для единицы комплектации** позволяет указать, что вес должен быть захвачен на основе количества, которое будет скомплектовано (например, три коробки). Обратите внимание, что для процессов комплектации строки производства и внутреннего перемещения средний вес будет использоваться, если используется параметр **Не регистрируется**.

Несколько методов захвата веса определены в политике обработки номенклатуры с учетом в двух единицах измерения. Каждый параметр метода захвата веса используется различными проводками. В следующей таблице перечислены параметры, используемые проводками.

| Метод                                     | Проводка                                |
|--------------------------------------------|--------------------------------------------|
| Способ исходящей регистрации веса           | Комплектация продаж, комплектация перемещения            |
| Способ регистрации веса при комплектации производства | Комплектация производства, потребление на производстве |
| Способ регистрации веса перемещения           | Проводка                                   |
| Когда регистрировать корректировку веса       | Корректировки, инвентаризация                      |
| Способ регистрации веса инвентаризации           | Инвентаризация                                   |
| Способ регистрации веса перемещения по складу | Перемещение по складу                         |

Чтобы ограничить процессы комплектации управления складом от регистрации весов, приводящих к корректировкам прибыли или убытка учета в двух единицах измерения, можно использовать метод расхождения исходящего веса. Метод отклонения исходящего веса применяется в следующих процессах для мобильных устройств: комплектация продаж, комплектация перемещение, комплектация производства, перемещения, инвентаризация и перемещения на складе. Можно использовать параметр **Ограничить расхождение веса**, если вес номенклатуры с учетом в двух единицах измерения не изменяется во время хранения на складе, и если не требуется корректировка прибылей и убытков по учету в двух единицах измерения. Можно использовать параметр **Разрешить отклонение веса**, если вес может изменяться и если корректировка по прибылям или убыткам по учету в двух единицах измерения требуется при записи изменения веса.

## <a name="unsupported-scenarios"></a>Неподдерживаемые сценарии

Не все workflow-процессы поддерживают обработку учета продуктов в двух единицах измерения с управлением склада. В настоящее время применяются следующие ограничения. Они применяются ко всем номенклатурам с учетом в двух единицах измерения независимо от того, есть ли у них теги.

### <a name="configuring-catch-weight-products-for-warehouse-management-processes"></a>Регистрация продуктов учета в двух единицах измерения для процессов управления складом

- Для продуктов с учетом в двух единицах измерения поддерживается только обработка формулы (не спецификация).
- Продукты с учетом в двух единицах измерения не могут быть связаны с группой аналитик отслеживания с помощью аналитики ответственного.
- Продукты с учетом в двух единицах измерения не могут использоваться как службы.
- Продукты с учетом в двух единицах измерения можно использовать в качестве "учитываемые в запасах продуктов" только в рамках группы моделей номенклатуры.
- Продукты с учетом в двух единицах измерения не могут использоваться вместе с функциональными возможностями для отслеживания "Активно в процессе продаж".
- Продукты с учетом в двух единицах измерения не могут использоваться вместе с функциональными возможностями для регистрации серийных номеров. Таким образом, продукты нельзя перенести из "пусто" в серийный номер в процессе комплектации/упаковки.
- Продукты с учетом в двух единицах измерения не могут использоваться вместе с функциональными возможностями для регистрации серийных номеров перед потреблением.
- Продукты с учетом в двух единицах измерения, для которых доступны варианты, не могут использоваться вместе с функциональной возможностью для преобразования единиц измерения вариантов.
- Продукт с учетом в двух единицах измерения не могут помечаться как коммерческий "комплект продуктов".
- Продукты с учетом в двух единицах измерения можно использовать только с группой номерных серий, которая имеет единицы обработки учета в двух единицах измерения, и которая имеет вторичную единицу измерения как самую низкую последовательность.
- Для продуктов с учетом в двух единицах измерения единицу складского учета можно преобразовать в вторичную единицу учета в двух единицах измерения только в том случае, если преобразование создает номинальное количество, которое превышает 1.
- Настройка штрих-кодов для продуктов с учетом в двух единицах измерения не поддерживает настройку переменного веса.

### <a name="order-processing"></a>Обработка заказа

- Создание предварительного уведомления об отгрузке (ASN/упаковочные структуры) не поддерживает сведения о весе.
- Количество заказа должно быть основано на единице учета в двух единицах измерения.

### <a name="inbound-warehouse-processing"></a>Входящая обработка на складе

- Получение грузомест требует, чтобы веса был присвоены во время регистрации, поскольку сведения о весе не поддерживаются как часть предварительного уведомления об отгрузке. Когда тег учета в двух единицах измерения используется, необходимо вручную назначить номер маркера на единицу учет в двух единицах измерения.
- Задание входящей проверки качества не поддерживается для продуктов с учетом в двух единицах измерения. Если это настроено, результаты проверки качества будут пропущены.

### <a name="inventory-and-warehouse-operations"></a>Операции запасов и складов

- Создание вручную карантинных заказов не поддерживается для продуктов с учетом в двух единицах измерения.
- Перемещение запасов вручную, которое связанно с открытой работой, не поддерживается для продуктов, учитываемых в двух единицах измерения.
- Загрузка грузоместа для инициализации запасов на складе не поддерживается для продуктов с учетом в двух единицах измерения.
- Процессы балансировки партий не поддерживаются для продуктов с учетом в двух единицах измерения.
- Обработка отрицательных физических остатков не поддерживается для продуктов с учетом в двух единицах измерения.
- Маркировка запасов не может использоваться для продуктов с учетом в двух единицах измерения.

### <a name="outbound-warehouse-processing"></a>Исходящая обработка на складе

- Функция комплектации кластера не поддерживается для продуктов с учетом в двух единицах измерения.
- Обработка комплектации и упаковки на складе не поддерживается для продуктов с учетом в двух единицах измерения.
- Для продуктов с учетом в двух единицах измерения работа, которая определена в шаблоне работы, не может запускаться автоматически.
- Для продуктов с учетом в двух единицах измерения ручная обработка на отборочной станции, когда работа комплектации упаковочного контейнера создается после закрытия контейнера, не поддерживается.
- Функция поштучного сканирования не поддерживается для продуктов с учетом в двух единицах измерения.

### <a name="production-processing"></a>Обработка производства

- Для продуктов с учетом в двух единицах измерения поддерживаются только заказы партий для продуктов по формуле.
- Функция канбан не поддерживается для продуктов с учетом в двух единицах измерения.
- Для продуктов с учетом в двух единицах измерения серийные номера не могут регистрироваться до потребления.
- Функция обращения грузомест из производства не поддерживается для продуктов с учетом в двух единицах измерения.
- Для продуктов с учетом в двух единицах измерения приемку нельзя зарегистрировать по серийному номеру.

### <a name="transportation-management-processing"></a>Обработка управления транспортировкой

- Обработка на рабочем месте формирования загрузки не поддерживается для продуктов с учетом в двух единицах измерения.
- Строки запросов на транспорт не поддерживаются для продуктов с учетом в двух единицах измерения.

### <a name="other-restrictions-and-behaviors-for-catch-weight-product-processing-with-warehouse-management"></a>Другие ограничения и поведения для обработки продуктов с учетом в двух единицах измерения при управлении складом

- Во время процессов комплектации, в которых пользователю не отображается подсказка для идентификации аналитик отслеживания, назначение веса осуществляется на основании среднего веса. Это поведение происходит, например, когда комбинация аналитик отслеживания используется в одном месте и, после обработки комплектации пользователем, только одно значение аналитики отслеживания остается в местонахождении.
- Когда запасы резервируются для продукта с учетом в двух единицах измерения, который настроен для процессов управления складом, резервирование выполняется на основании минимального веса, который определен, даже если это количество является последним количеством обработки в наличии. Это поведение отличается от поведения для номенклатур, которые не настроены для процессов управления складом. Имеется одно исключение для этого ограничения. Для комплектации производства, когда комплектуется последнее обрабатываемое количество продукта с учетом в двух единицах измерения с контролем по серийному номеру, используется фактический вес.
- Процессы, использующие вес как часть расчетах мощности (пороговые значения волны, максимальные рабочие перерывы, максимальные значения контейнера, загрузка производственных мощностей местоположения и т. д.), не используют фактический вес запасов. Вместо этого процессы основаны на весе физической обработки, который определяется для продукта.
- В общем случае, функции Commerce не поддерживаются для продуктов с учетом в двух единицах измерения.
- Для продуктов с учетом в двух единицах измерения состояние запасов невозможно обновить из **Изменение статуса склада**.

### <a name="catch-weight-tags"></a>Теги учета в двух единицах измерения

Тег учета в двух единицах измерения можно создать с помощью процесса приложения склада, создать вручную в форме или создать с помощью процесса информационного объекта. Если тег учета в двух единицах измерения связывается с входящей строкой исходного документа, например строкой заказа на покупку, тег будет зарегистрирован. Если строка используется для исходящей обработки, тег будет обновлен как доставленный.

В дополнение к ограничениям, которые в настоящее время применяются для продуктов с учетом в двух единицах измерения, продукты с учетом в двух единицах измерения и с тегами имеют другие ограничения помимо существующих в данное время.

- Все обновления запасов вручную (то есть обновления, которые не были выполнены с помощью мобильного устройства) должны включать соответствующие обновления вручную в соответствующих тегах учета в двух единицах измерения, так как эти обновления не выполняются автоматически. Например, журналы ручных корректировок обновляют запасы, но не соответствующие теги учета в двух единицах измерения.
- Необходимо вручную обновить теги учета в двух единицах измерения для отображения перемещений для пополнения. Это обусловлено тем, что система не может захватить вес при обработке работы по пополнению и вместо этого записывается средний вес.
- Смешанное грузоместо лицензии в настоящее время не поддерживается для номенклатур с учетом в двух единицах измерения и тегами.
- Обработка получения заказа на возврат для продаж может записывать теги учета в двух единицах измерения. Однако процесс не проверяет, что возвращенный тег совпадает с тегом, первоначально отгруженным для заказа на продажу.
- Пункт меню мобильного устройства с кодом действия **Регистрация потребления материалов** в настоящее время не поддерживает запись тегов учета в двух единицах измерения.
- Хотя процессы инвентаризации поддерживаются для номенклатур с учетом в двух единицах измерения и тегами, они ограничены. Например, можно использовать параметры мобильного устройства для инвентаризации номенклатуры с учетом в двух единицах измерения, и используется средний вес. Однако теги с учетом в двух единицах измерения не обновляются автоматически проводкой инвентаризации. После завершения проводки по инвентаризации теги с учетом в двух единицах измерения должны быть обновлены вручную, чтобы они соответствовали запасам. Если номенклатуры, которые не были первоначально в местоположении, подсчитываются в этом местоположении, используется номинальный вес.
- Консолидация грузоместа в настоящее время не поддерживает номенклатуры с учетом в двух единицах измерения и тегами.
- Функция сторнирования работы не поддерживается для номенклатур с учетом в двух единицах измерения и тегами, которые отслеживаются по номеру тега.

> [!NOTE]
> Предыдущая информация о тегах с учетом в двух единицах измерения действительна только в том случае, если у продукта с учетом в двух единицах измерения имеется полностью отслеживаемый метод отслеживания аналитики тегов с учетом в двух единицах измерения (то есть если параметр **Метод отслеживания аналитики тегов учета в двух единицах измерения** политики обработки номенклатур с учетом в двух единицах измерения задан как **Аналитики продукта,аналитики отслеживания и все аналитики хранения**). Если номенклатура с учетом в двух единицах измерения имеет частичное отслеживание по тегу (то есть, параметр **Метод отслеживания аналитики тегов учета в двух единицах измерения** политики обработки номенклатур с учетом в двух единицах измерения задан как **Аналитики продукта, аналитики отслеживания и статус запасов**), применяются дополнительные ограничения. Так в этом случае видимость между тегом и запасами теряется, некоторые дополнительные сценарии не поддерживаются.
