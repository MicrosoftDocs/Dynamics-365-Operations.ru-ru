# <a name="landed-cost-vs-transportation-management"></a>Стоимость на складе в сравнении с управлением транспортировкой

[!include [banner](../../includes/banner.md)]

Microsoft Dynamics 365 Supply Chain Management предоставляет два разных модуля для работы с транспортировкой: **управление транспортировкой** (TMS) и **стоимость на складе**. В этом разделе содержится сводная информация по функциям, используемым двумя модулями, и выделяются различия между ними. Можно использовать эту информацию, чтобы определить, какой модуль наилучшим образом подходит для бизнеса. Может оказаться, что некоторые бизнес-процессы работают лучше с TMS, тогда как другие лучше всего работают со стоимостью на складе. Затем, в зависимости от бизнес-требований, можно выбрать использование только одного модуля или объединить два модуля.

Этот раздел не является исчерпывающим обзором всех функций каждого модуля. Вместо этого в нем описывается доступная функциональность, поскольку она относится к транспортировке товаров от поставщика на склад вашей организации, где их можно потреблять.

## <a name="terminology-reference-data-and-reporting-differences"></a>Терминология, ссылочные данные и различия в отчетности

### <a name="terminology-comparison"></a>Сравнение терминологии

В TMS и стоимости на складе используются разные термины для похожих понятий. В следующей таблице перечислены эти термины и их использование в двух модулях.

| Термин TMS | Термин стоимости на складе | Основание |
|----------|------------------|-------|
| **Загрузить**<p>*Загрузка* — это совокупность отгрузок, которые переносятся одновременно в одну и ту же транспортную единицу (например, грузовик или судно). Загрузки могут быть либо входящими, либо исходящими.</p> | **Поездка**<p>Обычно *рейс* — это одно судно, которое перемещается по одному и тому же *рейсу*. Рейс может содержать несколько *исходящих контейнеров* и может также включать входящие заказы от различных юридических лиц вашей организации. Рейсы могут быть только входящими.</p> | TMS также использует термин *номер рейса*, который означает информационное поле, где можно ввести идентификатор. Однако ни одна из функций не связана с полем TMS, и оно не связано с понятием *рейс* в стоимости на складе. |
| **Маршрут**<p>*Маршрут* — это физический путь транспортировки из источника в место назначения.</p> | **Путь**<p>*Путь* — это физический путь транспортировки из источника в место назначения. Каждый рейс должен быть назначен для пути.</p> | |
| **Сегменты маршрута**<p>У маршрута может быть несколько *сегментов маршрута*, каждый из которых представляет этап пути. Маршрут может включать несколько перевозчиков или сервисов, каждый из которых считается сегментом маршрута.</p> | **Участки пути**<p>У каждого пути может быть несколько *отрезков*, каждый из которых представляет этап пути. Отрезок может быть частью транспортировки, обработки пошлины или другого действия.</p> | |
| **Контейнеры**<p>*Контейнером* может быть любой тип упаковки или упаковки, используемый TMS.</p> | **Грузовые контейнеры**<p>*Контейнер отгрузки* — это литерал, физический контейнер отгрузки, из контейнеровозов.</p> | |
| **Товарные коды**<p>В TMS (и в других местах Supply Chain Management) *товарные коды* определяют типы товаров. Они могут влиять на тарифы, обработку опасных материалов и т. д.</p> | **Товарные коды**<p>В стоимости на складе *товарные коды* задаются на уровне юридического лица. Они обычно используются только для отчетности.</p> | В стоимости на складе также могут использоваться товарные коды, которые используются для TMS и других мест в Supply Chain Management. Однако товарные коды стоимости на складе используются только стоимостью на складе. |

### <a name="some-reference-data-isnt-shared"></a>Общие ссылочные данные не являются общими

В TMS и стоимости на складе нет общих ссылочных данных для таких объектов, как настройка затрат, рейсы и отрезки. Если используются оба модуля, необходимо повторно создать необщие ссылочные данные.

### <a name="intercompany-reports-dont-work-with-goods-in-transit"></a><a name="intercompany-reports"></a>Внутрихолдинговые отчеты не работают с товарами в пути

Следующие отчеты не работают в сочетании с функцией товаров в пути в составе стоимости на складе:

- [Отчет по внутрихолдинговым итоговым значениям по товарам в пути](https://docs.microsoft.com/dynamicsax-2012/appuser-itpro/intercompany-goods-in-transit-totals-report-intercompanygoodsintransittotals)
- [Отчет по внутрихолдинговым итоговым значениям по товарам в пути](https://docs.microsoft.com/dynamicsax-2012/appuser-itpro/intercompany-goods-in-transit-totals-report-intercompanygoodsintransittotals)

В этих отчетах предполагается, что товары передаются в транспортировку, как только будет выпущена отборочная накладная продажи, и что они переносятся на в запасы из транспортировки после поступления. Однако товары в пути не обрабатываются таким образом. Поэтому при совместном использовании товаров в пути и внутрихолдинговых функций результаты обоих этих отчетов будут неверны.

## <a name="using-the-two-modules-together"></a>Использование двух модулей вместе

TMS можно использовать как для входящих, так и для исходящих операций. Хотя стоимость на складе предоставляет большинство основных функций, таких как TMS для входящих операций, система также добавляет некоторые функциональные возможности. Таким образом, может возникнуть необходимость использования TMS для исходящих операций и стоимости на складе для входящих операций.

В общем, мы не рекомендуем использовать оба модуля вместе для входящих операций. Следует использовать модуль, который наилучшим образом соответствует вашим требованиям. Если вы используете два модуля вместе, вам не придется совместно использовать исходные документы. Например, не следует использовать один и тот же заказ на покупку как для загрузки в TMS, так и для рейса в стоимости на складе. В частности, следует убедиться, что система не настроена на автоматическое создание входящих загрузок. Если номенклатуры из заказов на покупку включаются в рейс, они не могут обрабатываться как часть загрузки.

## <a name="goods-in-transit"></a>Товары в пути

Одной из основных функций стоимости на складе является возможность обработки товаров в пути. Обработка товаров в пути позволяет осуществлять финансовое владение отгруженными номенклатурами до того, как они физически получены на складе назначения. Обработка товаров в пути часто требуется для международной торговли.

### <a name="tms-goods-in-transit-features"></a>Функции товаров в пути TMS

TMS в настоящее время не поддерживает обработку товаров в пути.

### <a name="landed-cost-goods-in-transit-features"></a>Функции товаров в пути в стоимости на складе

Обработка товаров в пути является основной функцией стоимости на складе и обеспечивает следующие функциональные возможности:

- **Склады для товаров в пути** — склад для товаров в пути может быть связан с каждым складом в Supply Chain Management. Товары перемещаются на склады для товаров в пути после того, как будет выставлена накладная по исходному заказу на покупку. Физически зарезервированные номенклатуры становятся недоступными для потребления до тех пор, пока они не будут получены на своем физическом складе. Таким образом, можно получить финансовое владение товарами путем выставления накладной по заказу на покупку.
- **Счет ГК для товаров в пути** — стоимость на складе добавляет конкретный счет разноски главной книги в профиль разноски заказа на покупку для обработки товаров в пути. Этот счет главной книги для товаров в пути действует как счет типа "по товарам выставлены накладные, товары не получены". Когда по исходному заказу на покупку выставляется накладная и создается заказ для товаров в пути, прямая стоимость заказа на закупку разносится на счет ГК для товаров в пути до тех пор, пока товары не будут получены в целевом физическом местоположении.
- **Обновление управления отслеживанием** — для заказов товаров в пути поддерживается функция управления отслеживанием в стоимости на складе. Управление отслеживанием позволяет обновить ожидаемую дату прибытия товаров, пока они в пути. После этого управление отслеживанием обновляет рейс и связанные строки заказа на покупку. Ожидаемая дата поставки затем доступна для сводного планирования и логистики.
- **Активация чрезмерных/недостаточных проводок** — если между ожидаемыми товарами в строке рейса и фактическими товарами, полученными на складе, есть отличие, стоимость на складе разрешает его путем создания чрезмерных/недостаточных проводок. Затем можно управлять этими проводками на основе этого способа их обработки, используя либо заказ на покупку, либо журнал перемещения. Чрезмерные и недостаточные проводки обеспечивают связь с рейсом, заказом на покупку и новым журналом перемещения, или заказом на покупку для расхождения.
- **Заказы товаров в пути**— стоимость на складе представляет новый документ-источник, называемый *заказ товаров в пути*. Заказ товаров в пути позволяет выставить накладную по исходному заказу на покупку для получения финансового владения на номенклатуры. При выставлении накладной по заказу на покупку новый исходный документ создается для каждой строки заказа на покупку с комбинацией складских аналитик. Товары затем физически перемещаются на склад товаров в пути, где они физически зарезервированы по заказу товаров в пути и не могут быть потреблены, пока не получены заказы товаров в пути.

## <a name="miscellaneous-charges-and-shipment-costs"></a>Накладные расходы и затраты на отгрузку

Одним из важнейших аспектов отгрузки и управления отгрузкой товаров является распознавание накладных расходов, связанных с отгрузками. Эти накладные расходы не являются прямой стоимостью запасов, связанной с заказом на покупку и отгрузкой или рейсом. Вместо этого они представляют собой дополнительные затраты, которые вводятся с учетом сути движения товаров от поставщика к вашей организации. Эти затраты могут включать в себя затраты на транспортировку, связанные с отгрузкой товаров из одного физического местоположения в другое или затраты на пошлину, связанные с импортом товаров от иностранного поставщика.

Стоимость на складе обрабатывает эти затраты путем создания нового типа структуры затрат, который называется *автозатраты*. TMS обрабатывает эти затраты, используя функции накладных расходов.

### <a name="tms-charge-and-cost-features"></a>Функции накладных расходов и затрат в TMS

TMS использует накладные расходы для управления дополнительными затратами на загрузки, которые распределяются по связанным строкам документа-источника. Эти накладные расходы могут быть заданы на уровне либо в строке заказа на покупку, либо в заголовке заказа на покупку.

В TMS накладные расходы могут распределяться по весу, объему или количеству запасов. Расходы могут быть распределены по расходам на фрахт или дополнительные транспортные услуги. Дальнейшая разработка потребуется для использования дополнительных методов распределения в TMS.

### <a name="landed-cost-charge-and-cost-features"></a>Функции накладных расходов и затрат ы стоимости на складе

Стоимость на складе представляет собой набор функций затрат, которые обрабатывают дополнительные затраты, связанные с отгрузкой товаров. Эти затраты называются автоматическими затратами, они применяются во время первоначальной выдачи накладной для отгрузки с использованием суммы оценочной стоимости. Каждый тип затрат управляется в собственной разноске. После разноски фактических накладных для накладных расходов оценочные стоимости автоматически обновляются, чтобы они соответствовали фактическим затратам.

Кроме того, можно выставить накладные для накладных расходов, связанных с отгрузкой товаров, во время рейса из порта происхождения или в любое время после получения товаров. Эта возможность обеспечивает большую гибкость для потребления товаров.

В следующем списке описываются некоторые понятия для функций накладных расходов и затрат в стоимости на складе:

- **Область затрат** — затраты и накладные расходы могут быть назначены разным уровням или *областям затрат* в рейсе. Затраты могут быть применены на уровне рейса и разбиваются на каждую номенклатуру в рейсе. Кроме того, затраты могут быть применены на уровне контейнера, заказа на покупку, номенклатуры или заказа на перемещение. Все накладные расходы могут быть обработаны отдельно в различных областях и разбиты на стоимость по номенклатуре.
- **Методы распределения** — несколько методов распределения доступны в стоимости на складе. Накладные расходы могут быть распределены по количеству, сумме в рублях, значению, весу, объему, измерению или определяемому пользователем объемному параметру.
- **Управление клирингом/отклонением** — накладные расходы настраиваются в качестве своего типа кода затрат в стоимости на складе. Каждый тип кода затрат позволяет определить клиринговый счет для оценочных накладных расходов, а также счета начисления и отклонения по цене покупки для расхождений по оценочной стоимости и фактическим затратам. После первоначальной разноски оценочной стоимости есть кредит на клиринговый счет. После разноски фактических накладных фактические накладные расходы разносятся, а оценочная стоимость дебетуется из клирингового счета.

## <a name="matching-freight-bills-and-invoices"></a>Сопоставление векселей фрахта с накладными

### <a name="tms-bill-and-invoice-matching-features"></a>Функции сопоставления векселей и накладных TMS

TMS может сопоставлять векселя фрахта, содержащие оценочную стоимость и накладные, с фактическими затратами на фрахт. Накладные можно получить в электронном или бумажном формате. Сопоставление расхождения между оценочной стоимостью и фактическими затратами не влияет на оценку запасов.

Дополнительные сведения см. в разделе [Выверка фрахта в модуле "Управление транспортировкой"](../transportation/reconcile-freight-transportation-management.md).

### <a name="landed-cost-bill-and-invoice-matching-features"></a>Функции сопоставления векселей и накладных в стоимости на складе

Стоимость на складе может сопоставлять векселя фрахта для оценочных накладных расходов и может выставлять накладную для фактических накладных затрат на оценочную стоимость. В момент выставления накладной расходы на фрахт находятся в журнале накладных, а оценочная стоимость удаляется с клирингового счета. Кроме того, фактические накладные расходы разносятся по себестоимости проданных товаров для номенклатуры, вместе с расхождениями между расчетными расходами и фактическими расходами. Такое поведение позволяет обеспечить гибкость обработки накладных.

## <a name="tracking"></a>Отслеживание

Важной характеристикой как в TMS, так и в стоимости на складе является возможность отслеживания товаров и определения их местоположения в пути от точки происхождения до конечного склада. Отслеживание позволяет обновить местоположение товаров и ожидаемые сроки их прибытия на склад для потребления. В дополнение к статусу товаров в пути стоимость на складе предоставляет ожидаемые и фактические даты для каждого этапа рейса.

### <a name="tms-tracking-features"></a>Функции отслеживания TMS

TMS предоставляет ограниченные функции для отслеживания входящих загрузок. Система отображает даты и позволяет создавать интеграцию, чтобы найти точную позицию (например, на странице **Статус транспортировки**).

Для каждого сегмента маршрута можно ввести ожидаемые расписания и время прибытия.

### <a name="landed-cost-tracking-features"></a>Функции отслеживания в стоимости на складе

Стоимость на складе может предоставлять управление отслеживанием для каждого рейса, начиная с порта происхождения до конечного пункта назначения. Управление отслеживанием устанавливает статус рейса. Статус указывает, выполняется ли рейс, в таможне для проверки или в локальной доставке на пути к целевому складу. Статус может быть установлен на уровне строки заказа на покупку, контейнера и заголовка рейса. Таким образом, реализован более детализированный контроль.

Кроме того, подтвержденная ожидаемая дата, которая основана на указанном времени возможной сделки, связана с каждым этапом пути. Подтвержденные и ожидаемые даты поставки добавляются в строку заказа на покупку и в заказы товаров в пути и могут использоваться для сводного планирования и логистики. В дополнение к ожидаемым датам, фактические даты могут быть обновлены. Последующие этапы в пути затем будут соответствующим образом обновлены.

## <a name="multi-leg-journeys"></a>Пути с несколькими участками пути

Понятие пути определяет, где товары находятся в процессе, и управляет статусом товаров, когда они в пути. Товары могут проходить несколько отрезков пути, и можно связать различные затраты с определенным отрезком. Примерами являются расход на фрахт судна в пути и местные транспортные затраты по транспортировке товаров из порта до пункта назначения.

### <a name="tms-multi-leg-journey-features"></a>Функции пути из нескольких отрезков в TMS

В TMS маршруты могут быть разбиты на сегменты маршрута для представления путей с несколькими отрезками. TMS может распределять наличные курсы и дополнительные расходы для отдельных сегментов маршрута.

Дополнительные сведения см. в разделе [Планирование маршрутов фрахта с несколькими остановками](../transportation/plan-freight-transportation-routes-multiple-stops.md).

### <a name="landed-cost-multi-leg-journey-features"></a>Функции пути из нескольких отрезков в стоимости на складе

Стоимость на складе предоставляет платформу для перемещения товаров из точки происхождения до пункта назначения через последовательность отрезков, представляющих собой остановки или этапы пути. Отрезки комбинируются для создания шаблонов путей, которые определяют порядок перемещения товаров от поставщика в вашу организацию.

В каждом случае пути можно дополнительно определить статус каждой строки заказа на покупку и связанных с ней значений времени возможной сделки. Общее время упреждения для всех отрезков используется для определения ожидаемой даты доставки. Однако для применения путей с несколькими отрезками требуется обработка товаров в пути. Путь товаров в пути управляется в таблице, которая относится к стоимости на складе.

## <a name="receiving-by-container"></a>Получение по контейнеру

Важно управлять разделением и хранением товаров на судне. На судне товары могут храниться в одном контейнере или в нескольких контейнерах. При получении товаров они получаются в контейнерах, и разные контейнеры в пути могут быть получены в разное время или разные даты.

Как TMS, так и стоимость на складе обеспечивают функции управления поступлением товаров в контейнере. TMS использует концепцию загрузок для управления товарами, заказами на покупку и заказами на перемещение, связанными с контейнером отгрузки. TMS поддерживает поступление на основе упаковочной структуры, полученной с помощью предварительного уведомления об отгрузке (ASN). Стоимость на складе использует концепцию контейнеров отгрузки для обработки заказов на покупку и управления накладными расходами, связанными с контейнером на судне.

### <a name="tms-receiving-by-container-features"></a>Функции получения по контейнеру в TMS

TMS поддерживает входящие ASN, все варианты получения через мобильное приложение управления складом и все методы получения через клиент Supply Chain Management.

### <a name="landed-cost-receiving-by-container-features"></a>Функции получения по контейнеру в стоимости по складу

Для поддержки получения по контейнеру стоимость на складе создает записи контейнеров отгрузки и связывает заказы на покупку с конкретным контейнером отгрузки, используя его код контейнера. Накладные расходы затем могут быть применены к контейнеру отгрузки и разбиты, чтобы они были связаны с соответствующими заказами на покупку.

Контейнеры в стоимости на складе могут быть получены с помощью нового типа чека, который называется *чеком товаров в пути*, с помощью журналов прибытия или через получение на мобильных устройствах. При использовании журналов прибытия количества могут быть инициализированы из заказа товаров в пути или в строках исходного заказа на покупку в контейнере. Стоимость на складе предоставляет два вида работ для получения через мобильное приложение управления складом.

Стоимость на складе не предоставляет ASN для электронного поступления товаров. Кроме того, эта система не поддерживает потоки мобильного приложения управления складом, которые обрабатывают получение, получение грузоместа или получение смешанных грузомест.

## <a name="rate-shopping-by-vendor"></a>Подбор ставок по поставщику

Подбор ставок позволяет компании выбирать поставщика транспортных услуг на основе наименьшей цены, самого быстрого маршрута или комбинации обоих.

### <a name="tms-rate-shopping-features"></a>Возможности подбора ставок в TMS

TMS позволяет определить поставщика и решения маршрута для входящих и исходящих заказов. Например, можно определить самый быстрый маршрут или наименьшую ставку для отгрузки.

TMS обеспечивает полный подбор ставок и оптимизацию фрахта с помощью рабочего места ставок и маршрутов, гибких параметров ставок с помощью механизма ставок, API механизма ставок для интеграции с внешними сторонами и поддержки объемного веса.

Дополнительные сведения см. в разделе [Системы управления транспортировкой](../transportation/transportation-management-engines.md).

### <a name="landed-cost-rate-shopping-features"></a>Функции подбора ставок в стоимости на складе

Стоимость на складе содержит только ограниченную поддержку подбора ставок по поставщику. Хотя можно ввести значения для экспедитора груза, стоимость на складе не сравнивает их между несколькими поставщиками.

## <a name="driver-check-incheck-out-with-appointment-scheduling"></a>Регистрация отметки прибытия и отъезда водителя с планированием встреч

Функции регистрации отметки прибытия и отъезда водителя позволяют системе отслеживать прибытия и предотвращать заторы на погрузочной площадке.

### <a name="tms-driver-check-incheck-out-features"></a>Возможности регистрации отметки прибытия и отъезда водителя в TMS

TMS позволяет создавать *встречи*. Встреча представляет события, которые происходят в дебаркадере для приемки заказа на покупку, отгрузки заказа на продажу или обработки входящей либо исходящей загрузки в определенные дату и время. Встречи гарантируют, что дебаркадеры доступны для погрузки и выгрузки товаров, и они помогают предотвратить ситуации, когда несколько перевозчиков прибывают в одну и ту же точку одновременно.

Система поддерживает разрешение водителям регистрацию прибытия в определенных воротах дебаркадера.

### <a name="landed-cost-driver-check-incheck-out-features"></a>Возможности регистрации отметки прибытия и отъезда водителя в стоимости по складу

Стоимость по складу может хранить ожидаемые даты и время, когда контейнер будет доставлен.

## <a name="transfer-orders-and-additional-costs"></a>Заказы на перемещение и дополнительные затраты

Часто у компаний должна быть возможность перемещать товары между складами. Когда происходит такое перемещение, с ним связаны затраты, и вам может потребоваться распознавать такие затраты. В стоимости по складу заказы на перемещение могут быть добавлены в рейс или на судно для отслеживания перемещения товаров с одного склада или объекта на другой, оценки времени доставки и ожидаемой даты доставки, а также добавления накладных расходов для перемещения товаров.

### <a name="tms-transfer-order-cost-features"></a>Функции затрат по заказам на перемещение в TMS

TMS поддерживает создание расходов за фрахт, связанных с перемещениями. Хотя эти расходы можно просмотреть в заказе на перемещение, стоимость на складе не добавляется к затратам по номенклатуре. Выверка фрахта поддерживается путем создания векселя фрахту на основе таких расходов. Вексель фрахта затем сопоставляется с накладной по фрахту поставщика.

### <a name="landed-cost-transfer-order-cost-features"></a>Функции затрат по заказам на перемещение в стоимости на складе

Стоимость на складе позволяет добавлять заказы на перемещение на судно или в рейс. Таким образом, можно добавлять затраты на отгрузку в рейс в качестве сопоставления запасов во время поступления заказа на перемещение. По накладным расходам можно выставить накладные по фактическим затратам и отслеживать по рейсу, чтобы обновить себестоимость проданных товаров. Хотя заказы на перемещение не используют обработку товаров в пути в стандартной версии, их можно добавлять в рейсы. Стоимость на складе добавляется к общим затратам по каждой номенклатуре.
