# <a name="action-messages"></a>Сообщения по действию

[!include [banner](../includes/banner.md)]

Сообщение по действию — это созданное системой предложение изменить существующий спланированный, утвержденный или подтвержденный заказ.

Сообщения по действию создаются расчетом сводного планирования в ответ на изменение потребностей. Например, дата или количество отгрузки изменяются в заказе на продажу после того, как вы уже создали заказ на покупку, чтобы выполнить требование для этого заказа на продажу. В этом случае расчет сводного планирования создает одно или несколько сообщений по действиям, которые предлагают вам обновить заказ на покупку. Вы решаете, нужно ли выполнять предлагаемые изменения.

Можно настроить, как сообщения по действиям рассчитываются для группы покрытия, присоединенной к номенклатуре.

## <a name="select-action-messages"></a>Выбор сообщений по действиям

На странице **Группы покрытия** можно выбрать сообщения по действиям, которые система должна создать, и группы покрытия или номенклатуры, к которым применяются сообщения. В следующей таблице перечислены сообщения действия, которое вы можете выбрать.

| Сообщение | Описание |
|---|---|
| Аванс | По мере необходимости система создает сообщения действия, чтобы переместить заказ на более раннюю дату. В поле **Резерв ускорения** можно указать максимальное число дней, которое может пройти между приходом и выдачей без действия ускорения. |
| Отложить | По мере необходимости система создает сообщения действия, чтобы переместить заказ на более позднюю дату. В поле **Резерв отсрочки** можно указать максимальное число дней, которое может пройти между приходом и выдачей без действия отсрочки. |
| Сокращение | Система будет создавать сообщения действий, когда производственные заказы, заказы на покупку и другие проводки по приходу должны быть уменьшены для предотвращения избыточного уровня запасов. |
| Увеличение | Система будет создавать сообщения действий, когда производственные заказы, заказы на покупку и другие проводки по приходу должны быть увеличены для предотвращения недостатка запасов. |
| Производные действия | Сообщения по действиям, созданные для проводок прихода, будут переданы любым производным требованиям, и необходимые сообщения по действиям будут создаваться для проводок прихода, удовлетворяющих этим требованиям. |

В следующих разделах представлено несколько подробных сценариев.

## <a name="increase-and-decrease-actions-with-product-default-order-quantities"></a>Действия по увеличению и уменьшению с количествами по умолчанию для заказа продуктов

На странице **Параметры заказа по умолчанию** для номенклатуры можно настроить минимальное количество заказа, максимальное количество заказа и несколько значений. Затем система использует эти настройки при предложении действий, чтобы гарантировать, что эти предложения никогда не приведут к появлению недостаточных поставок.

Например, имеется номенклатура со следующими настройками на странице **Параметры заказа по умолчанию**:

- **Минимальное количество по заказу:** *0*
- **Максимальное количество по заказу:** *90*
- **Кратный:** *20*

Если имеется спрос на количество 60 данной номенклатуры, сводное планирование создаст спланированный заказ на покупку для количества 60. Если спрос увеличивается на 30, сводное планирование предложит увеличение на 40, поскольку оно будет учитывать значение, кратное 20, и никогда не вызовет недостаточные поставки.

## <a name="action-messages-for-orders-related-to-safety-stock"></a>Сообщения по действиям для заказов, связанных с резервным запасом

Сообщения по действиям для заказов, поставляющих резервный запас, никогда не предложат уменьшить количество меньше количества, которое требуется для резервного запаса. Другими словами, если имеется заказ, предоставляющий резервный запас и потребность клиента, а спрос уменьшается или отменяется, сводное планирование будет предлагать уменьшение спланированного заказа на снизившийся спрос. Однако никогда не будет предлагаться значение, которое ниже необходимого для резервного запаса.

Система не будет предлагать отсрочку действий для поставки резервного запаса, поскольку предполагается, что резервный запас должен быть поставлен в требуемое время и на требуемую дату.

### <a name="advance-and-postpone-actions-related-to-boms"></a>Ускорение и откладывание действий, связанных со спецификациями

Действия, которые относятся к компонентам спецификаций (BOM), должны применяться до действий их родительских номенклатур, поскольку это может повлиять на другие заказы, связанные с спецификациями верхнего уровня. Затем необходимо снова запустить сводное планирование, чтобы пересчитать и предложить соответствующие действия.

Например, есть следующая ситуация:

- Окончательный товар *FG* типа *производство* имеет спецификацию, которая включает сырья *RM*.
- Сегодняшняя дата — 21 января.
- Существующий выпущенный производственный заказ для *FG* запланирован на 25 января.
- Для поддержки существующего производственного заказа в сводном планировании создан спланированный заказ на покупку на требуемое сырье *RM*. Этот заказ имеет требуемую дату 25 января.
- Новый заказ на продажу для *FG* создается сегодня. Он имеет дату требования сегодня (21 января).
- Дата 21 января будет закрыта для поставки в календаре *RM*, но дата 22 января открыта.

При запуске сводного планирования создаются сообщения действий ускорения, которые предлагают переместить вверх ранее запланированное производство, чтобы можно было выполнить текущий заказ.

- Для удовлетворения нового спроса предлагается переместить производственный заказ на *FG* до 21 января. (Это предложение делается без учета закрытой даты для *RM*.)
- Поскольку сырье *RM* все еще необходимо для производственного заказа, предлагается также перемещение вверх спланированного заказа на покупку. Однако на этот раз проверяется календарь *RM*. Поэтому предлагается переместить спланированный заказ на покупку для *RM* на 22 января (так как дата 21 января закрыта).

Как можно видеть, требуемое сырье *RM* теперь прибудет слишком поздно для запланированного производства *FG*. Поэтому сначала необходимо применить действие ускорения к спланированному заказу на покупку для сырья *RM*, затем снова запустите сводное планирование. В этот момент сводное планирование будет создавать новое сообщение действия, предлагающее переместить производственный заказ *FG* на 22 января.
