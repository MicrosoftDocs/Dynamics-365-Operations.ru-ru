# <a name="troubleshoot-discrete-manufacturing"></a>Устранение неполадок дискретного производства

В этом разделе описывается устранение проблем, которые могут встретиться при работе с дискретным производством.

## <a name="i-receive-the-following-error-message-warehouse-management-processes-are-currently-being-used-if-work-lines-are-not-yet-registered-you-can-cancel-the-created-work-and-any-load-or-shipment-lines-and-then-continue-with-the-picking-or-shipping-process"></a>Получено следующее сообщение об ошибке: "Процессы управления складом в настоящее время используются. Если строки работ еще не зарегистрированы, можно отменить созданную работу и все строки загрузки или отгрузки, а затем продолжить процесс комплектации или отгрузки."

### <a name="issue-description"></a>Описание проблемы

Эта проблема возникает при попытке зарезервировать или выпустить работу для строки, но складская проводка имеет статус *Скомплектовано*, что указывает на то, что материал скомплектован.

### <a name="issue-resolution"></a>Устранение проблемы

Чтобы устранить эту проблему, выполните один из следующих шагов.

- Измените статус производственного заказа обратно на значение *Оценено* или *Выпущено*.
- Откройте страницу сведений для соответствующего производственного заказа. В области действий на вкладке **Склад** в группе **Остановить** выберите **Остановить и отменить комплектацию**, чтобы отменить комплектацию всех скомплектованных проводок. Затем выберите **Удалить остановку** для обработки производственного заказа.

Вот объяснение функций *Разукомплектовать* и *Остановить*:

- **Разукомплектовать** — эта функция обращает статус складских проводок для строк спецификации и строк формулы, имеющих статус от *Скомплектовано* до *Заказано*. Когда работа для комплектации сырья завершена, статус строк устанавливается в *Скомплектовано*. Этот статус предотвращает сброс производственного заказа до статуса *Создано*. В этом случае можно использовать функцию *Разукомплектовать* для возврата проводок из статуса *Скомплектовано*, а затем сбросить производственный заказ в статус *Создано*.
- **Остановить** — эта функция устанавливает флаг **Остановлено** для производственного заказа, чтобы запретить любое обновление статуса в заказе. Можно найти флаг **Остановлено** на экспресс-вкладке **Склад** на странице сведений о производственном заказе.

> [!NOTE]
>
> - Кнопки отображаются только в том случае, если производственный заказ создается для номенклатур, включенных для складских процессов.
> - Группа **Остановить** видна только на вкладке **Склад** в области действия страницы сведений о производственном заказе. Она не отображается на экспресс-вкладке **Склад** страницы списка **Производственные заказы**.

## <a name="the-matching-resource-name-isnt-updated-after-i-change-a-worker-name-in-the-global-address-book"></a>Соответствующее имя ресурса не обновляется после изменения имени работника в глобальной адресной книге.

### <a name="issue-description"></a>Описание проблемы

Если изменить имя работника в глобальной адресной книге, имя соответствующего ресурса не обновляется с шаблоне группы ресурсов.

### <a name="issue-resolution"></a>Устранение проблемы

Этот сценарий в настоящий момент не поддерживается. Чтобы устранить эту проблему, необходимо вручную обновить имя ресурса.

## <a name="when-i-create-a-new-production-order-i-dont-receive-the-following-message-insert-the-active-version-of-bill-of-material-and-route"></a>При создании нового производственного заказа не выводится следующее сообщение: "Вставить активную версию спецификации и маршрута?"

### <a name="issue-description"></a>Описание проблемы

При создании нового производственного заказа после ввода номера номенклатуры поля **Сайт** и **Склад** автоматически устанавливаются в сайт и склад по умолчанию, которые определены на странице **Параметры заказов по умолчанию** для готовой номенклатуры товаров. Кроме того, автоматически вводятся активный номер спецификации и номер маршрута. Не выводится следующее сообщение, предлагающее ввести эти значения:

> Вставить активную версию спецификации и маршрута?

### <a name="issue-resolution"></a>Устранение проблемы

Вам не предлагается вставить номера спецификаций и маршрутов, если выбран продукт, для которого определен сайт и склад на странице **Параметры заказа по умолчанию**. Запрос выводится только в том случае, если эти значения не определены для выбранного продукта.

## <a name="production-orders-arent-shown-on-the-marking-page"></a>Производственные заказы не отображаются на странице маркировки.

### <a name="issue-description"></a>Описание проблемы

Некоторые производственные заказы не отображаются на странице **Маркировка**.

### <a name="issue-resolution"></a>Устранение проблемы

Продукты, имеющие следующую конфигурацию, недоступны для маркировки. Поэтому они не будут отображаться на странице **Маркировка**:

- Продукты определяются как продукты типа *учет в двух единицах измерения*.
- Они включены для дополнительных складских процессов.
- Они настраиваются для управления с помощью принципа *Стандартные затраты*.

## <a name="when-i-try-to-end-a-production-order-i-receive-the-following-error-message-calculating-bom-consumptioncost-value-must-be-negative-upon-issue-from-inventory"></a>При попытке завершить производственный заказ появляется следующее сообщение об ошибке: "Расчет потребления спецификации/значение себестоимости должно быть отрицательным при выпуске из запасов."

Эта проблема была исправлена в версии 10.0.15.

## <a name="when-the-status-of-a-production-order-is-changed-from-reported-as-finished-to-end-i-receive-the-following-error-messages-update-conflict-the-standard-cost-does-not-match-with-the-financial-inventory-value-after-the-update-and-a-critical-error-has-occurred-in-function-inventcostmovementcheckvariance"></a>Когда статус производственного заказа изменяется с "Принято" на "Окончание", я получаю следующее сообщение об ошибке: "Конфликт при обновлении. В результате обновления стандартные затраты не соответствуют финансовой величине запасов" и "Критическая ошибка в функции InventCostMovement.checkVariance."

Эта проблема возникает из-за того, что базовые данные были изменены другим процессом. Процесс будет пытаться обновлять данные до пяти раз. Если конфликт по-прежнему существует после пяти попыток, будут получены следующие сообщения об ошибке:

> Конфликт при обновлении. В результате обновления стандартные затраты не соответствуют финансовой величине запасов.

> Критическая ошибка в функции InventCostMovement.checkVariance.

Такое поведение предусмотрено разработчиками. Чтобы устранить эту проблему, возобновите пакетное задание. Оно должно завершить выполнение.

Если после возобновления пакетного задания происходят постоянные сбои, убедитесь, что точность округления валюты по умолчанию в главной книге соответствует округлению, применяемому к значениям в таблице InventSum. Если точность округления была изменена на несоответствующее значение, то, возможно, необходимо изменить его обратно на соответствующее значение. Найдите **ModifiedDateTime**. В этом случае значение обычно показывает, что точность округления была недавно изменена.

## <a name="when-i-release-to-a-warehouse-i-receive-the-following-error-message-item-rm-could-not-be-fully-reserved-ensure-that-the-full-quantity-is-available-or-reserve-the-items-manually-if-the-reservation-field-on-the-bom-line-is-set-to-manual-or-started-could-not-release-the-order-to-warehouse-because-some-materials-could-not-be-reserved-however-the-status-is-updated-to-released"></a>При выпуске на склад появляется следующее сообщение об ошибке: "Не удается полностью зарезервировать номенклатуру RM. Убедитесь, что все количество доступно, или зарезервируйте номенклатуры вручную, если поле резервирования в строке спецификации имеет значение "Вручную" или "Запущено". Не удалось выпустить заказ на склад, так как не удалось зарезервировать часть материалов." Однако статус обновляется до "Выпущено".

### <a name="issue-description"></a>Описание проблемы

Если не все номенклатуры строки спецификации являются физически доступными, когда выпущен производственный заказ, а в качестве политики **Выпуск на склад** задано значение *Требуется полное резервирование* в производственном заказе, будет выведено следующее сообщение об ошибке:

> Не удается полностью зарезервировать номенклатуру RM. Убедитесь, что все количество доступно, или зарезервируйте номенклатуры вручную, если поле резервирования в строке спецификации имеет значение "Вручную" или "Запущено". Не удалось выпустить заказ на склад, так как не удалось зарезервировать часть материалов.

### <a name="issue-resolution"></a>Устранение проблемы

Такое поведение является намеренным и работает как ожидается.

## <a name="when-i-try-to-end-a-production-order-and-report-as-finished-i-receive-the-following-error-message-total-good-quantity-reported-as-finished-for-the-production-will-be-1-feedback-for-the-last-operation-is-000-in-total"></a>При попытке завершить производственный заказ и выполнить приемку появляется следующее сообщение об ошибке: "Общее количество товара, учтенного как готовый для производства, будет %1. Обратная связь для последней операции составляет 0,00 (итог)."

### <a name="issue-description"></a>Описание проблемы

При попытке разнести журнал приемки по производственному заказу появляется следующее сообщение об ошибке:

> Общее количество товара, учтенного как готовый для производства, будет %1. Обратная связь для последней операции составляет 0,00 (итог).

### <a name="possible-cause"></a>Возможная причина

Эта проблема возникает в том случае, если количества, которые были разнесены в последних операциях, были неверны. Например, если производство запущено, но количество, которое должно быть запущено, не распределено, журнал карты маршрута будет разноситься без каких-либо строк. Чтобы подтвердить ситуацию, откройте производственный заказ, а затем на панели операций на вкладке **Вид** выберите **Карта маршрута**.

### <a name="workaround"></a>Обходное решение

Можно устранить эту проблему путем разноски дополнительных журналов для операций, для которых журналы не были корректно разнесены. Перезапустите производственный заказ и выберите, чтобы разнести дополнительные журналы. Это действие не запускает производственный заказ, но ведет к разноске журналов. Затем карта маршрута отображает количества, которые были разнесены, и вы должны успешно завершить производственные заказы.

## <a name="can-i-report-a-production-order-as-finished-while-i-report-the-error-quantity-but-not-while-i-report-the-time-or-material-quantity"></a>Можно ли сообщить о завершении производственного заказа, когда я сообщаю о количестве брака, но не когда я сообщаю о времени или количестве материала?

Невозможно сообщить о количестве брака в производственном заказе, если не указано также количество товаров. Этот сценарий **не** поддерживается. Обновление приемки в конечном итоге приведет к сбою при попытке завершения производственного заказа, и появится следующее сообщение об ошибке:

> Отсутствует количество, готовое к приемке.

## <a name="can-i-trace-the-serial-numbers-of-finished-goods-against-the-serial-numbers-of-consumed-goods"></a>Можно ли отслеживать серийные номера готовых изделий по серийным номерам потребленных товаров?

Вы не можете отслеживать серийные номера готовых изделий по серийным номерам материалов, которые потребляются производственным заказом, чтобы сделать эти готовые товары. Этот сценарий в настоящий момент не поддерживается. Для решения проблемы можно создать производственные заказы на количество, равное 1.
