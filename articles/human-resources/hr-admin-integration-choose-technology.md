# <a name="choose-a-data-integration-technology"></a>Выбор технологии интеграции данных

Dynamics 365 Human Resources управляет бизнес-данными, которые полезны в различных бизнес-процессах. В этой статье представлено руководство по различным вариантам интеграции с данными, управляемыми модулем Human Resources, с описанием характеристик различных интеграционных технологий, чтобы интеграторы могли принимать обоснованные решения относительно того, какие технологии лучше всего подходят их нужд.

## <a name="data-integration-vision"></a>Представление об интеграции данных

Microsoft видит бизнес-данные как ключевой актив, делающий вашу компанию уникальной. Данные вашей компании являются очень ценными. Данные, собираемые и поддерживаемые одной частью бизнеса, относятся к данным, собранным другой частью бизнеса, и эти отношения могут использоваться для улучшения бизнес-процессов и бизнес-аналитики в организации. Обеспечение простого, безопасного и стабильного доступа к бизнес-данным, независимо от того, какая система "владеет" данными, являются ключевым принципом, связанным с концепцией, для интеграции данных с модулем Human Resources.

Исторически интеграция данных между несколькими системами была сложной.
Корпорация Майкрософт предпринимает шаги для упрощения интеграции данных, а большой шаг к этой цели реализован с помощью [Common Data Service](https://docs.microsoft.com/powerapps/maker/common-data-service/data-platform-intro).

В дальнейшем Human Resources делает Common Data Service предпочтительным общим интерфейсом для данных Human Resources. Со временем предполагается, что все наиболее важные данные, управляемые модулем Human Resources, будут представлены в Common Data Service. Мы рекомендуем использовать Common Data Service в качестве технологии, выбранной для большинства систем интеграции приложений. Хотя мы понимаем, что не все данные, которые могут потребоваться приложению, в настоящее время присутствуют в Common Data Service, и что на временных шкалах проекта может потребоваться альтернативная технология, сообщите нам когда Common Data Service не соответствует вашим потребностям в интеграции.

## <a name="integration-technologies"></a>Технологии интеграции

В следующих разделах описываются разнообразные технологии интеграции данных, доступные для использования с модулем Human Resources.

### <a name="common-data-service-entities"></a>Объекты Common Data Service

Common Data Service — является предпочтительным общедоступным интерфейсом данных для модуля Human Resources. Common Data Service основано на зрелой платформе, которая выросла из платформы Dynamics 365 "XRM", на которой создаются решения [Dynamics 365 Customer Engagement](https://docs.microsoft.com/dynamics365/#pivot=business-apps&panel=customer-engagement).

Common Data Service предоставляет платформу для объектов данных и API для доступа к этим объектам. При развертывании в организации модуля Human Resources он связывается с экземпляром Common Data Service, и объекты, поддерживающие данные Human Resources, развертываются в этот экземпляр Common Data Service, делая объекты и их данные доступными для любого приложения, которое может подключиться к экземпляру Common Data Service. В зависимости от того, какие приложения модуля Human Resources используются, модуль Human Resources производит операции непосредственно с объектами Common Data Service (это случай Attract и Onboard) или синхронизирует данные с этими объектами Common Data Service.

После того, как объекты данных с данными, которые нужны для интегрируемых приложений, будут представлены в Common Data Service, вы можете полностью использовать [Common Data Service и API, которые поддерживает](https://docs.microsoft.com/powerapps/#pivot=home&panel=developer). Среди поддерживаемых API используется [Веб-API Microsoft Dynamics 365](https://docs.microsoft.com/dynamics365/customer-engagement/developer/use-microsoft-dynamics-365-web-api), обеспечивающий реализацию OData для доступа к данным Common Data Service.

Объекты Common Data Service и соответствующие API являются оптимальным вариантом для доступа к данным Human Resources из веб-приложений, веб-служб/API и других приложений, подключающихся к каналам OData.

> [!NOTE]
> Если к решению о том, чтобы сделать Common Data Service предпочтительным интерфейсом данных для Human Resources, вы пришли недавно, вы можете обнаружить, что необходимые для интеграции объекты данных Human Resources еще не доступны в Common Data Service<sup>1</sup>. Если объекты Human Resources, необходимые для вашей интеграции, еще недоступны, необходимо подождать, пока объекты данных станут доступны, либо воспользоваться одной из других интеграционных технологий, описанных ниже.
> 
> По умолчанию интеграция Common Data Service отключена в новых средах, не содержащих предоставленных демонстрационных данных. Он включается в новых средах, содержащих демонстрационные данные, а синхронизация данных начинается при подготовке среды. После того как среда готова к синхронизации данных, можно включить интеграцию.

<sup>1</sup>Список объектов Human Resources доступен в Common Data Service, см. [Human Resources и Common Data Service](https://docs.microsoft.com/dynamics365/unified-operations/talent/corehrentities). Для Attract и Onboard все объекты доступны в Common Data Service.

### <a name="dmfdixf-entities"></a>Объекты DMF/DIXF

Система Human Resources, разработанная в основном на той же платформе, что и приложения Finance and Operations, предоставляет [платформу управления данными (DMF)](https://docs.microsoft.com/dynamics365/unified-operations/dev-itpro/data-entities/data-entities-data-packages?toc=/fin-and-ops/toc.json), которая иногда называется структурой импорта и экспорта данных (DIXF), и набор сущностей данных, которые могут использоваться для импорта/экспорта данных в/из модуля Human Resources. Хотя объекты Common Data Service являются предпочтительным интерфейсом интеграции данных для Human Resources, DMF-объекты будут все еще полезны в некоторых обстоятельствах, например:

- Объекты Common Data Service еще не доступны.

- Для интеграции требуется высокая производительность с поддержкой массового импорта и экспорта данных.

Объекты DMF в настоящее время обеспечивают наиболее полное покрытие данных для данных модуля Human Resources.

DMF не подходит для интеграции в реальном времени (например, когда требуется немедленная обратная связь пользователя в интерфейсе пользователя), так как операции пакета планируются как пакетные задания и зачастую будут иметь задержку в 1-2 минуты перед тем, как служба пакета забирает задание для выполнения, а также время задержки, необходимое для завершения операции импорта/экспорта.

DMF может быть наилучшим вариантом, если требуется высокая пропускная способность (например, запланирован импорт/экспорт большого числа тысяч записей в ночное время).

> [!NOTE]
> DMF недоступно для Attract и Onboard.

### <a name="dmf-package-rest-api"></a>Пакет DMF REST API

DMF предоставляет [REST API](https://docs.microsoft.com/dynamics365/unified-operations/dev-itpro/data-entities/data-management-api) для управления пакетами данных. Этот API может использоваться для программного взаимодействия с DMF, что позволяет выполнять следующие действия:

- Импорт пакета данных.

- Экспорт пакета данных.

- Проверка статуса операции импорта/экспорта.

Пакет DMF REST API полностью поддерживается в Human Resources: Core HR.

### <a name="azure-sql-db-byod"></a>Azure SQL DB (BYOD)

DMF дополнительно предоставляет мощную функцию (обычно именуемую [Использование своей собственной базы данных](https://docs.microsoft.com/dynamics365/unified-operations/dev-itpro/analytics/export-entities-to-your-own-database), или BYOD), которая позволяет Human Resources осуществлять экспорт данных в собственную базу данных Microsoft Azure SQL. Это обеспечивает огромную гибкость, так как когда данные представлены в собственной базе данных SQL, можно использовать любые приложения или промежуточное программное обеспечение, которые могут подключаться к SQL-хранилищу данных.

BYOD обычно рассматривается как решение только для чтения. Несмотря на то что вы можете обрабатывать и сохранять любые данные, которые необходимо использовать в базе данных SQL Azure (например, для гибридных данных), данные, хранящиеся в базе данных SQL Azure, не будут синхронизированы обратно с модулем Human Resources.

BYOD подходит для систем отчетности, интеграции данных, гибридных данных в качестве источника данных для конвейера [фабрики данных Azure](https://docs.microsoft.com/azure/data-factory/).

> [!NOTE]
> BYOD недоступно для Attract и Onboard.

### <a name="odata-enabled-entities"></a>Объекты, поддерживающие OData

Для большинства объектов DMF также разрешен доступ с помощью службы управления Human Resources (OData). Документация, предоставленная для [службы OData Finance and Operations](https://docs.microsoft.com/dynamics365/unified-operations/dev-itpro/data-entities/odata), обычно применяется также к Human Resources, но документация по созданию собственных объектов, предоставляемых OData, не будет применяться.

Хотя реализация Common Data Service и OData, предоставляемая Common Data Service (посредством [Веб-API Microsoft Dynamics 365](https://docs.microsoft.com/previous-versions/dynamicscrm-2016/developers-guide/mt593051(v=crm.8))), является предпочтительной по отношению к службе данных Human Resources, служба данных Human Resources в настоящее время обладает более полным покрытием объектов для данных Human Resources.

### <a name="excel-add-in"></a>Надстройка Excel

[Надстройка Excel](https://docs.microsoft.com/dynamics365/unified-operations/dev-itpro/office-integration/use-excel-add-in?toc=/dynamics365/unified-operations/talent/toc.json) использует объекты на основе OData, расположенные ниже поверхности. Она позволяет конечному пользователю получать и изменять данные Human Resources с помощью привычного пользовательского интерфейса Excel.

Надстройка Excel подходит для нерегламентированного импорта или экспорта данных экспертами по бизнес-доменам. Для выполнения повторяющейся интеграции данных, для которой требуется программная автоматизация, более подходящим будет другая технология интеграции.

### <a name="data-integrator"></a>Интегратор данных

Администратор Common Data Service предоставляет [службу интегратора данных](https://docs.microsoft.com/powerapps/administrator/data-integrator), которую можно использовать для интеграции данных с Common Data Service. Интеграторы данных могут использоваться для определения интеграционных проектов (часто на основе заранее определенных шаблонов, которые разработчики приложений адаптируют к определенным интеграциям). Проекты интеграции могут быть запланированы для автоматического запуска с повторяющимся расписанием или выполняться вручную.

Проекты интеграторов данных подходят для пакетной интеграции Common Data Service, а также для обеспечения удобного выбора для интеграции между приложениями семейства Dynamics 365. В качестве примера Корпорация Майкрософт предоставляет готовый шаблон интегратора данных, который может использоваться для интеграции данных из Human Resources в Dynamics 365 Finance. Дополнительные сведения см. в разделе [Интеграция из Dynamics 365 Human Resources в Dynamics 365 Finance](hr-admin-integration-finance.md).

### <a name="power-query"></a>Power Query

Интегратор данных также поддерживает [Power Query](https://docs.microsoft.com/power-query/power-query-what-is-power-query) (через [функцию расширенного запроса](https://docs.microsoft.com/powerapps/administrator/data-integrator#advanced-data-transformation-and-filtering)).
Power Query обеспечивает мощную, гибкую фильтрацию данных и преобразование, включая язык формул Rich M, и, скорее всего, знаком тем, кто имеет опыт разработки отчетов Power BI.

## <a name="deciding-on-an-integration-technology"></a>Выбор технологии интеграции

Благодаря такому множеству доступных технологий интеграции выбор используемого в данный подхода интеграции может иногда быть большим. С течением времени, особенно в случае взросления покрытия данных в Common Data Service, решение становится проще, в то время как в большинстве случаев будет предпочтительнее интерфейс данных Common Data Service. Но до тех пор может оказаться, что это Common Data Service не соответствует вашим потребностям. В следующей таблице приведены основные характеристики различных параметров технологии интеграции.

| Технология/инструмент/API    | Повторяющиеся интеграции                   | Синхронный/асинхронный                    | Программный доступ через API        | Соответствующие тома данных                                   | Покрытие данных                       |
|------------------------|------------------------------------------|---------------------------------------------|-------------------------------------------|------------------------------------------------------------|-------------------------------------|
| Объекты Common Data Service           | Да, используется интегратор данных или промежуточное ПО | Синхронно-асинхронно, пакеты (через интегратор данных) | Да, через Веб-API Microsoft Dynamics 365 (OData) | Разное, в зависимости от случая (поддерживает разбиение по страницам для интерактивного использования) | Улучшение<sup>2</sup>                       |
| Объекты DMF           | Да, по графику через промежуточное ПО        | Асинхронно, пакеты                                | Да, через пакет DMF REST API         | Высокий (сотни тысяч записей)                    | Высокое                                |
| Пакет DMF REST API   | Да, по графику через промежуточное ПО        | Асинхронно, пакеты                                | Да                                       | Высокий (сотни тысяч записей)                    | API поддерживает все объекты DMF       |
| BYOD                   | Да, по графику администратора в Human Resources        | Асинхронно, пакеты                                | Нет<sup>3</sup>                                    | Высокий (сотни тысяч записей)                    | Поддерживает все объекты DMF           |
| Объекты, поддерживающие OData | Да, с промежуточным ПО                    | Синхронизация                                        | Да, через службу данных Human Resources (OData)  | Разное, в зависимости от случая (поддерживает разбиение по страницам для интерактивного использования) | Высокое                                |
| Надстройка Excel           | Нет                                       | Синхронизация                                        | Нет                                        | Средний (десятки тысяч записей)                      | Поддержка всех объектов с поддержкой OData |
| Интегратор данных        | Да, запланировано в интеграторе данных        | Асинхронно, пакеты                                | Нет                                        | В зависимости от варианта использования                                       | Поддерживает все объекты Common Data Service           |

<sup>2</sup>Корпорация Майкрософт в значительной мере увеличивает покрытие данных для объектов Common Data Service и рекомендует использовать Common Data Service как предпочтительный интерфейс данных при наличии покрытия. В настоящее время покрытие данных Common Data Service относительно низкое для объектов на основе DMF и OData.

<sup>3</sup>К базе данных SQL можно получить программный доступ.

## <a name="summary"></a>Сводка

Бизнес-данные являются ценным активом, но его ценность может быть значительно снижена, если необходимо использовать данные для конкретной цели (например, для создания отчетов, гибридных данных или пользовательских приложений). Dynamics 365 Human Resources предоставляет несколько технологий для работы с данными вне пользовательского интерфейса приложения Human Resources, позволяя интегрировать доступ приложений к данным. В этом разделе описаны доступные технологии интеграции и некоторые их ключевые характеристики. Эти сведения помогут принять более качественные решения о том, какой подход использовать для проектов интеграции.
